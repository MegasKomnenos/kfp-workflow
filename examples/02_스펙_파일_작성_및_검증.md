# 02. 스펙 파일 작성 및 검증

kfp-workflow의 모든 동작은 YAML 스펙 파일로 정의됩니다. 이 튜토리얼에서는 `PipelineSpec`, `BenchmarkSpec`, `ServingSpec`, `TuneSpec`의 구조와 검증 방법을 설명합니다.

## PipelineSpec (학습 파이프라인 스펙)

위치: `configs/pipelines/`

### 전체 구조

```yaml
# configs/pipelines/my_experiment.yaml

# ── 메타데이터 ──────────────────────────────────────
metadata:
  name: my-experiment           # 파이프라인 이름 (필수)
  description: "실험 설명"       # 설명 (선택)
  version: v1                   # 버전 태그 (선택, 기본값: v1)

# ── 런타임 환경 ─────────────────────────────────────
runtime:
  namespace: kubeflow-user-example-com   # K8s 네임스페이스
  image: kfp-workflow:latest             # 파이프라인 컨테이너 이미지
  image_pull_policy: IfNotPresent        # 이미지 풀 정책
  service_account: default-editor        # K8s 서비스 어카운트
  host: "http://127.0.0.1:8888"         # KFP API 호스트
  port_forward_namespace: kubeflow       # 포트포워딩 네임스페이스
  port_forward_service: svc/ml-pipeline  # 포트포워딩 서비스
  use_gpu: false                         # GPU 사용 여부
  torch_num_threads: 4                   # PyTorch CPU 스레드 수
  resources:                             # 컴퓨팅 리소스
    cpu_request: "4"
    cpu_limit: "4"
    memory_request: 8Gi
    memory_limit: 8Gi
    gpu_request: "0"
    gpu_limit: "0"

# ── 스토리지 ────────────────────────────────────────
storage:
  data_pvc: dataset-store        # 데이터 PVC 이름
  model_pvc: model-store         # 모델 PVC 이름
  storage_class: local-path      # 스토리지 클래스
  data_size: 8Gi                 # 데이터 PVC 크기
  model_size: 8Gi                # 모델 PVC 크기
  data_mount_path: /mnt/data     # 데이터 마운트 경로
  data_subpath: ""               # 선택: PVC 내 데이터 하위 경로
  model_mount_path: /mnt/models  # 모델 마운트 경로

# ── 모델 설정 ──────────────────────────────────────
model:
  name: mambasl-cmapss           # 플러그인 이름 (필수)
  version: v1                    # 모델 버전
  config:                        # 플러그인별 아키텍처 파라미터
    d_model: 32                  # 모델 차원
    d_state: 8                   # 상태 차원
    d_conv: 3                    # 컨볼루션 커널 크기
    expand: 1                    # 확장 비율
    window_size: 30              # 슬라이딩 윈도우 크기
    max_rul: 125.0               # 최대 RUL 값

# ── 데이터셋 설정 ──────────────────────────────────
dataset:
  name: cmapss                   # 데이터셋 이름 (필수)
  version: v1
  config:                        # 플러그인별 데이터셋 파라미터
    fd_name: FD001        # C-MAPSS 데이터셋 (FD001-FD004)
    download_policy: if_missing  # 다운로드 정책
    feature_mode: settings_plus_sensors  # 특성 모드
    norm_mode: condition_minmax  # 정규화 모드

# ── 학습 하이퍼파라미터 ────────────────────────────
train:
  seed: 42                       # 난수 시드
  batch_size: 64                 # 배치 크기
  learning_rate: 0.001           # 학습률
  weight_decay: 0.0001           # 가중치 감쇠
  max_epochs: 2                  # 최대 에포크
  patience: 1                    # 조기종료 인내 횟수
  val_split: 0.2                 # 검증 데이터 비율
  selection_metric: rmse         # 모델 선택 기준 메트릭
  score_weight: 0.01             # 하이브리드 메트릭 가중치
```

### 필드 설명

| 섹션 | 설명 | 필수 여부 |
|------|------|----------|
| `metadata` | 파이프라인 이름 및 버전 | 필수 |
| `runtime` | 실행 환경 (네임스페이스, 이미지, 리소스) | 필수 |
| `storage` | PVC 스토리지 설정 | 선택 (기본값 제공) |
| `model` | 모델 플러그인 및 아키텍처 설정 | 필수 |
| `dataset` | 데이터셋 참조 및 설정 | 필수 |
| `train` | 학습 하이퍼파라미터 | 선택 (기본값 제공) |

### model.config와 dataset.config

이 두 필드는 `Dict[str, Any]` 타입으로, 플러그인별로 자유롭게 정의됩니다. mambasl-cmapss 플러그인의 경우:

**model.config 지원 필드:**
- `d_model`, `d_state`, `d_conv`, `expand` — 모델 아키텍처
- `num_kernels`, `tv_dt`, `tv_B`, `tv_C`, `use_D` — Mamba 블록 설정
- `projection`, `dropout`, `huber_delta` — 학습 관련
- `window_size`, `max_rul` — 데이터 처리

**dataset.config 지원 필드:**
- `fd_name` — C-MAPSS 데이터셋 (FD001, FD002, FD003, FD004), 스칼라 문자열
- `download_policy` — 플러그인별 다운로드 정책. C-MAPSS 플러그인은 `never`, `if_missing`, `always` 지원
- `feature_mode` — `settings_plus_sensors` 또는 `sensors_only`
- `norm_mode` — `condition_minmax`, `global_standard`, `condition_standard`

`spec validate`는 플러그인 스키마까지 함께 검증합니다. 잘못된 `model.config`, `dataset.config`, `train` 값은 경고가 아니라 즉시 실패합니다.

## ServingSpec (추론 서빙 스펙)

위치: `configs/serving/`

```yaml
# configs/serving/my_serving.yaml

metadata:
  name: my-model-serving          # InferenceService 이름
  description: "서빙 설명"
  version: v1

namespace: kubeflow-user-example-com  # 배포 네임스페이스
model_name: mambasl-cmapss            # 모델 플러그인 이름
model_version: v1                     # 모델 버전
model_pvc: model-store                # 모델 PVC 이름
model_subpath: mambasl-cmapss/v1      # PVC 내 모델 디렉토리 경로
runtime: custom                       # 런타임 타입
predictor_image: kfp-workflow:latest  # 커스텀 런타임 이미지
replicas: 1                           # 레플리카 수

resources:
  cpu_request: "2"
  cpu_limit: "2"
  memory_request: 4Gi
  memory_limit: 4Gi
  gpu_request: "0"
  gpu_limit: "0"
```

### 런타임 타입

| 타입 | 설명 |
|------|------|
| `custom` | 플러그인이 선택한 모델 파일을 로드하고 예측하는 커스텀 예측기 |
| `kserve-torchserve` | KServe 표준 TorchServe 런타임 |

`runtime: custom`인 경우 `predictor_image`가 필요하며, `metadata.name`은 `<name>-predictor-<namespace>`가 63자를 넘지 않아야 합니다.

## BenchmarkSpec (벤치마크 워크플로우 스펙)

위치: `configs/benchmarks/`

`BenchmarkSpec`은 세 가지를 한 번에 정의합니다:
- **model** — 벤치마크 동안 임시 배포할 `InferenceService`
- **scenario** — 데이터셋 정의 + 요청 파이프라인
- **metrics** — 시나리오 실행 중 수집할 메트릭 목록

```yaml
# configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml

metadata:
  name: mambasl-cmapss-benchmark-smoke
  description: MambaSL C-MAPSS benchmark smoke test with Kepler energy capture
  version: v1

runtime:
  namespace: kubeflow-user-example-com
  image: kfp-workflow:latest
  image_pull_policy: IfNotPresent
  service_account: default-editor
  host: "http://127.0.0.1:8888"
  port_forward_namespace: kubeflow
  port_forward_service: svc/ml-pipeline

storage:
  data_pvc: dataset-store
  model_pvc: model-store
  results_pvc: benchmark-store
  storage_class: local-path
  data_size: 8Gi
  model_size: 8Gi
  results_size: 8Gi
  data_mount_path: /mnt/data
  model_mount_path: /mnt/models
  results_mount_path: /mnt/benchmarks

model:
  model_name: mambasl-cmapss
  model_version: v1
  model_pvc: model-store
  model_subpath: mambasl-cmapss/v1
  predictor_image: kfp-workflow:latest
  service_name: mbsl-bench-smoke
  cleanup: true
  wait_timeout: 300

scenario:
  ref: scenarios/mambasl_cmapss_timeseries_smoke.yaml

metrics:
  - ref: metrics/kepler_predictor_energy.yaml
```

### 주요 필드

| 섹션 | 설명 | 필수 여부 |
|------|------|----------|
| `metadata` | 벤치마크 이름 및 버전 | 필수 |
| `runtime` | 실행 네임스페이스, 이미지, KFP 연결 정보 | 필수 |
| `storage` | 데이터/모델/결과 PVC 설정 | 선택 (기본값 제공) |
| `model` | 임시 KServe 배포 대상 모델 정보 | 필수 |
| `scenario` | 시나리오 정의 또는 ref | 필수 |
| `metrics` | 메트릭 정의 또는 ref 목록 | 선택 |

### scenario / metrics ref

`BenchmarkSpec`은 scenario와 metrics를 별도 YAML 또는 Python 심볼로 분리할 수 있습니다.

**YAML ref:**
```yaml
scenario:
  ref: scenarios/mambasl_cmapss_timeseries_smoke.yaml

metrics:
  - ref: metrics/kepler_predictor_energy.yaml
```

**Python ref:**
```yaml
scenario:
  ref: benchmark_defs/my_scenario.py:SCENARIO

metrics:
  - ref: benchmark_defs/my_metric.py:METRIC
```

또는 dataset / pipeline / metric 노드에서 `kind: python`과 `ref:`를 직접 사용할 수도 있습니다. `spec validate --type benchmark`는 ref를 실제로 해석하고, YAML은 inline으로 병합하며, Python은 표준 인터페이스 계약에 맞는지까지 검증합니다.

## TuneSpec (하이퍼파라미터 튜닝 스펙)

위치: `configs/tuning/`

TuneSpec은 PipelineSpec과 동일한 구조에 `hpo` 섹션이 추가됩니다:

```yaml
# configs/tuning/my_tune.yaml

# ── PipelineSpec과 동일한 섹션들 ─────────────────────
metadata:
  name: my-model-tune
  description: "HPO 튜닝 실험"

runtime:
  namespace: kubeflow-user-example-com
  image: kfp-workflow:latest

model:
  name: mambasl-cmapss
  config:
    d_model: 32
    d_state: 8

dataset:
  name: cmapss
  config:
    fd_name: FD001

train:
  seed: 42
  batch_size: 64
  max_epochs: 50
  selection_metric: rmse

# ── HPO 설정 (TuneSpec 전용) ─────────────────────────
hpo:
  algorithm: tpe                # 탐색 알고리즘 (tpe / random / grid)
  max_trials: 20                # 최대 시행 횟수
  max_failed_trials: 3          # 연속 실패 허용 횟수
  parallel_trials: 1            # 병렬 시행 수
  builtin_profile: default      # 내장 탐색 공간 (default / aggressive / custom)
  search_space: []              # custom 프로파일 사용 시 직접 정의
```

### hpo 섹션 필드 설명

| 필드 | 타입 | 기본값 | 설명 |
|------|------|--------|------|
| `algorithm` | str | `tpe` | 탐색 알고리즘 (`tpe`, `random`, `grid`) |
| `max_trials` | int | `20` | 최대 시행 횟수 |
| `max_failed_trials` | int | `3` | 연속 실패 시 조기종료 임계값 |
| `parallel_trials` | int | `1` | 동시 시행 수 (Katib용) |
| `builtin_profile` | str | `default` | 플러그인 내장 탐색 공간 프로파일 |
| `search_space` | list | `[]` | 커스텀 탐색 공간 (builtin_profile 무시) |

### 커스텀 탐색 공간 정의

`builtin_profile: custom`으로 설정하고 `search_space`에 직접 파라미터를 정의할 수 있습니다:

```yaml
hpo:
  algorithm: tpe
  max_trials: 30
  builtin_profile: custom
  search_space:
    - name: train.learning_rate
      type: log_float
      low: 0.0001
      high: 0.01
    - name: model.config.d_model
      type: categorical
      values: [32, 64, 128]
    - name: train.batch_size
      type: int
      low: 16
      high: 128
      step: 16
```

파라미터 타입:

| 타입 | 설명 | 필수 필드 |
|------|------|----------|
| `categorical` | 후보 값 목록에서 선택 | `values` |
| `int` | 정수 범위 | `low`, `high`, (`step` 선택) |
| `float` | 실수 범위 | `low`, `high`, (`step` 선택) |
| `log_float` | 로그 스케일 실수 범위 | `low`, `high` |

> 자세한 내용은 [11. 하이퍼파라미터 튜닝](11_하이퍼파라미터_튜닝.md)을 참고하세요.

## 스펙 검증

### 기본 검증

```bash
# 파이프라인 스펙 검증
kfp-workflow spec validate --spec configs/pipelines/mambasl_cmapss_smoke.yaml

# 벤치마크 스펙 검증
kfp-workflow spec validate --spec configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml --type benchmark

# 서빙 스펙 검증
kfp-workflow spec validate --spec configs/serving/mambasl_cmapss_serve.yaml --type serving

# 튜닝 스펙 검증
kfp-workflow spec validate --spec configs/tuning/mambasl_cmapss_tune.yaml --type tune
```

성공 시 출력:
```
Spec 'mambasl-cmapss-smoke' validated successfully.
{
  "metadata": { ... },
  "runtime": { ... },
  ...
}
```

### 오버라이드와 함께 검증

`--set` 플래그로 값을 변경한 후의 스펙도 검증할 수 있습니다:

```bash
kfp-workflow spec validate \
    --spec configs/pipelines/mambasl_cmapss_smoke.yaml \
    --set train.max_epochs=100 \
    --set model.config.d_model=128
```

### 일반적인 검증 오류

**필수 필드 누락:**
```
ValidationError: 1 validation error for PipelineSpec
metadata -> name
  field required
```
→ `metadata.name`을 추가하세요.

**잘못된 타입:**
```
ValidationError: 1 validation error for PipelineSpec
train -> batch_size
  value is not a valid integer
```
→ `batch_size`에 정수 값을 사용하세요.

**알 수 없는 플러그인:**
플러그인 검증은 `spec validate` 시점이 아닌 파이프라인 실행 시에 발생합니다.
등록된 플러그인은 `src/kfp_workflow/plugins/__init__.py`에서 확인할 수 있습니다.
