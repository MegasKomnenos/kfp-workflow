# 11. 하이퍼파라미터 튜닝

## 개요

kfp-workflow는 프로젝트 레벨의 HPO(Hyperparameter Optimization) 엔진을 내장하고 있습니다. Optuna 기반의 로컬 튜닝과 Kubeflow Katib 기반의 분산 튜닝을 모두 지원하며, CLI에서 알고리즘, 탐색 공간, 트라이얼 수 등 핵심 변수를 명시적으로 제어할 수 있습니다.

### 아키텍처 원칙

HPO 오케스트레이션은 **프로젝트가 소유**합니다. 플러그인은 프로젝트가 정의한 인터페이스에 따라 탐색 공간과 단일 트라이얼 목적 함수만 제공합니다.

```
┌─────────────────────────────────────────────┐
│           kfp-workflow (프로젝트)              │
│                                             │
│  tune/engine.py   ← Optuna 스터디 생성/실행   │
│  tune/katib.py    ← Katib 매니페스트 생성     │
│  specs.py         ← TuneSpec, SearchParamSpec │
│  plugins/base.py  ← HPO 계약 메서드           │
└──────────────────┬──────────────────────────┘
                   │ 플러그인이 구현
┌──────────────────▼──────────────────────────┐
│       mambasl-cmapss 플러그인 (어댑터)        │
│                                             │
│  hpo_search_space()  → 탐색 공간 반환        │
│  hpo_base_config()   → 기본 설정 반환        │
│  hpo_objective()     → 단일 트라이얼 실행     │
└─────────────────────────────────────────────┘
```

**핵심 설계**: 프로젝트의 `tune/engine.py`가 Optuna 스터디를 생성하고, 파라미터를 제안하고, 트라이얼 루프를 실행합니다. 플러그인은 Optuna `trial` 객체를 절대 직접 다루지 않습니다.

## TuneSpec (튜닝 스펙)

### 구조

`TuneSpec`은 `PipelineSpec`과 동일한 필드에 `hpo` 섹션을 추가한 것입니다:

```yaml
# configs/tuning/mambasl_cmapss_tune.yaml

metadata:
  name: mambasl-cmapss-hpo
  description: MambaSL C-MAPSS FD001 하이퍼파라미터 최적화
  version: v1

runtime:
  namespace: kubeflow-user-example-com
  image: kfp-workflow:latest
  use_gpu: false
  torch_num_threads: 4

storage:
  data_pvc: dataset-store
  model_pvc: model-store
  data_mount_path: /mnt/data
  model_mount_path: /mnt/models

model:
  name: mambasl-cmapss        # 플러그인 이름
  version: v1
  config: {}                  # hpo_base_config()에서 기본값 자동 제공

dataset:
  name: cmapss
  version: v1
  config:
    fd:
      - fd_name: FD001
    feature_mode: settings_plus_sensors
    norm_mode: condition_minmax

train:
  seed: 42
  max_epochs: 25              # HPO 트라이얼당 에포크 수
  patience: 5
  selection_metric: rmse
  score_weight: 0.01

# ── HPO 설정 (TuneSpec 전용) ──────────────────────
hpo:
  algorithm: tpe               # 탐색 알고리즘: tpe | random | grid
  max_trials: 20               # 최대 트라이얼 수
  max_failed_trials: 3         # 최대 실패 허용 수
  parallel_trials: 1           # 병렬 트라이얼 수
  builtin_profile: default     # 내장 탐색 공간: default | aggressive | custom
  # search_space: []           # 비어 있으면 builtin_profile 사용
```

### hpo 섹션 필드

| 필드 | 타입 | 기본값 | 설명 |
|------|------|--------|------|
| `algorithm` | string | `tpe` | 탐색 알고리즘 (`tpe`, `random`, `grid`) |
| `max_trials` | int | `20` | 최대 트라이얼 수 |
| `max_failed_trials` | int | `3` | 이 수만큼 실패하면 스터디 중단 |
| `parallel_trials` | int | `1` | 병렬 트라이얼 수 (Katib 전용) |
| `builtin_profile` | string | `default` | 내장 탐색 공간 프로필 |
| `search_space` | list | `[]` | 커스텀 탐색 공간 (비어 있으면 프로필 사용) |

### 탐색 알고리즘

| 알고리즘 | 설명 | 적합한 경우 |
|----------|------|------------|
| `tpe` | Tree-structured Parzen Estimator | 일반적 사용 (기본값). 이전 결과를 활용한 베이지안 최적화 |
| `random` | 무작위 탐색 | 탐색 공간이 넓거나 초기 탐색 시. 병렬 실행에 유리 |
| `grid` | 그리드 탐색 | categorical 파라미터만 있을 때. 완전 탐색 보장 |

## CLI 명령어

### `tune show-space` — 탐색 공간 미리보기

HPO를 실행하기 전에 어떤 파라미터가 탐색되는지 확인합니다:

```bash
# 기본 프로필의 탐색 공간 확인
kfp-workflow tune show-space \
    --spec configs/tuning/mambasl_cmapss_tune.yaml
```

출력:
```
                     Search Space
┏━━━━━━━━━━━━━━┳━━━━━━━━━━━━━┳━━━━━━━━━━━━━━━━━━━━━━━┓
┃ NAME         ┃ TYPE        ┃ RANGE / VALUES        ┃
┡━━━━━━━━━━━━━━╇━━━━━━━━━━━━━╇━━━━━━━━━━━━━━━━━━━━━━━┩
│ d_model      │ categorical │ 32, 64, 128           │
│ d_state      │ categorical │ 8, 16, 32             │
│ d_conv       │ categorical │ 3, 4                  │
│ lr           │ log_float   │ [0.0003, 0.003] (log) │
│ ...          │ ...         │ ...                   │
└──────────────┴─────────────┴───────────────────────┘
```

```bash
# aggressive 프로필로 전환
kfp-workflow tune show-space \
    --spec configs/tuning/mambasl_cmapss_tune.yaml \
    --set hpo.builtin_profile=aggressive

# JSON 출력 (스크립팅용)
kfp-workflow --json tune show-space \
    --spec configs/tuning/mambasl_cmapss_tune.yaml
```

### `tune run` — 로컬 Optuna HPO 실행

로컬 환경에서 Optuna 기반 하이퍼파라미터 탐색을 실행합니다:

```bash
# 기본 실행
kfp-workflow tune run \
    --spec configs/tuning/mambasl_cmapss_tune.yaml \
    --data-mount-path ./data

# 알고리즘 및 트라이얼 수 명시적 제어
kfp-workflow tune run \
    --spec configs/tuning/mambasl_cmapss_tune.yaml \
    --set hpo.algorithm=tpe \
    --set hpo.max_trials=30 \
    --data-mount-path ./data

# 결과를 파일로 저장
kfp-workflow tune run \
    --spec configs/tuning/mambasl_cmapss_tune.yaml \
    --data-mount-path ./data \
    --output results/best_params.json

# 스모크 테스트 (2 트라이얼, 2 에포크)
kfp-workflow tune run \
    --spec configs/tuning/mambasl_cmapss_tune.yaml \
    --set hpo.max_trials=2 \
    --set train.max_epochs=2 \
    --data-mount-path ./data
```

출력 예시:
```
  Best value   12.3456
  Trials       18/20 completed
  Pruned       2
  Failed       0
  Wall time    345.2s

Best parameters:
{
  "d_model": 64,
  "d_state": 16,
  "lr": 0.0012,
  "batch_size": 128,
  ...
}
```

### `tune katib` — 분산 Katib HPO

Kubeflow Katib Experiment 매니페스트를 생성하거나 클러스터에 적용합니다:

```bash
# 매니페스트 미리보기 (적용하지 않음)
kfp-workflow tune katib \
    --spec configs/tuning/mambasl_cmapss_tune.yaml \
    --dry-run

# 매니페스트를 파일로 저장
kfp-workflow tune katib \
    --spec configs/tuning/mambasl_cmapss_tune.yaml \
    --output kubeflow/katib/mambasl-hpo.yaml \
    --dry-run

# 클러스터에 직접 제출
kfp-workflow tune katib \
    --spec configs/tuning/mambasl_cmapss_tune.yaml
```

Katib는 Kubernetes 위에서 트라이얼을 분산 실행합니다. `parallel_trials` 설정으로 동시 실행 수를 제어할 수 있습니다.

## 탐색 공간

### 내장 프로필

플러그인은 두 가지 내장 탐색 공간 프로필을 제공합니다:

**default 프로필** — 핵심 파라미터 17개, 보수적인 범위:
```bash
kfp-workflow tune show-space --spec configs/tuning/mambasl_cmapss_tune.yaml
```

**aggressive 프로필** — 확장된 범위, 더 넓은 탐색:
```bash
kfp-workflow tune show-space \
    --spec configs/tuning/mambasl_cmapss_tune.yaml \
    --set hpo.builtin_profile=aggressive
```

주요 차이:

| 파라미터 | default | aggressive |
|----------|---------|------------|
| `d_conv` | 3, 4 | 2, 3, 4 |
| `dropout` | 0.0–0.3 | 0.0–0.4 |
| `window_size` | 30, 40, 50 | 30, 40, 50, 60, 70 |
| `max_rul` | 125, 130, 150 | 115, 120, 125, 130, 150 |

### 커스텀 탐색 공간

`hpo.search_space` 필드에 직접 파라미터를 정의하면 내장 프로필 대신 커스텀 탐색 공간을 사용합니다:

```yaml
hpo:
  builtin_profile: custom
  search_space:
    - name: d_model
      type: categorical
      values: [64, 128, 256]

    - name: lr
      type: log_float
      low: 0.0001
      high: 0.01

    - name: batch_size
      type: categorical
      values: [32, 64, 128]

    - name: dropout
      type: float
      low: 0.0
      high: 0.5
      step: 0.1
```

### 파라미터 타입

| 타입 | 설명 | 필수 필드 | 예시 |
|------|------|----------|------|
| `categorical` | 이산 값 목록에서 선택 | `values` | `values: [32, 64, 128]` |
| `int` | 정수 범위 | `low`, `high` | `low: 1, high: 10` |
| `float` | 실수 범위 (균등) | `low`, `high` | `low: 0.0, high: 0.5` |
| `log_float` | 실수 범위 (로그 스케일) | `low`, `high` | `low: 0.0001, high: 0.01` |

> `log_float`는 학습률처럼 크기 차수(order of magnitude)가 중요한 파라미터에 적합합니다.

### CLI에서 탐색 공간 오버라이드

`--set` 플래그로 탐색 공간의 개별 요소도 오버라이드할 수 있습니다:

```bash
# 프로필 변경
--set hpo.builtin_profile=aggressive

# 알고리즘 변경
--set hpo.algorithm=random

# 트라이얼 수 변경
--set hpo.max_trials=50
```

HPO 결과에서 얻은 최적 파라미터를 `--set`으로 전달하여 전체 학습을 실행합니다:

    --spec configs/pipelines/mambasl_cmapss_smoke.yaml \

## 데이터 캐싱

`hpo_objective` 메서드는 트라이얼 간 데이터를 캐싱합니다:

- `window_size`와 `max_rul`이 탐색 공간에 **포함되지 않으면** — 데이터 로드 및 전처리가 한 번만 실행됨 (빠름)
- `window_size`와 `max_rul`이 탐색 공간에 **포함되면** — 값이 변경될 때마다 재전처리 필요 (느림)

> 빠른 HPO가 필요하면 `window_size`와 `max_rul`을 고정하고 탐색 공간에서 제외하는 것을 권장합니다.

## TrialPruned 예외

플러그인의 `hpo_objective()` 메서드는 트라이얼을 중단해야 할 때 프로젝트 레벨의 `TrialPruned` 예외를 발생시킵니다:

```python
from kfp_workflow.tune.exceptions import TrialPruned

def hpo_objective(self, spec, params, data_mount_path):
    # 데이터가 부족한 경우
    if len(x_train) < params["batch_size"]:
        raise TrialPruned()
    # ... 학습 실행 ...
```

이 예외는 `kfp_workflow.tune.exceptions`에 정의되어 있으며, Optuna 타입이 플러그인 코드에 누출되지 않도록 합니다.
