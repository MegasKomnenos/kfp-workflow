# 03. 파이프라인 컴파일 및 제출

## 개요

kfp-workflow의 파이프라인 워크플로우는 두 단계로 구성됩니다:

1. **컴파일** — YAML 스펙을 KFP v2 파이프라인 패키지(YAML)로 변환
2. **제출** — 컴파일된 파이프라인을 Kubeflow에 제출하여 실행

같은 패턴이 벤치마크 워크플로우에도 적용됩니다. 학습은 `pipeline compile` / `pipeline submit`, 벤치마크는 `benchmark compile` / `benchmark submit`을 사용하며, 실행 모니터링은 둘 다 `pipeline run *` 명령으로 공통 처리합니다.

## 파이프라인 DAG

컴파일된 파이프라인은 다음 5개의 컴포넌트로 구성됩니다:

```
load_data → preprocess → train → evaluate → save_model
```

각 컴포넌트는:
- 동일한 Docker 이미지(`kfp-workflow:latest`)에서 실행
- 전체 `PipelineSpec` JSON을 입력으로 받음
- `ModelPlugin`에 작업을 위임
- PVC를 마운트하여 데이터/모델 공유

## 파이프라인 컴파일

### 기본 컴파일

```bash
kfp-workflow pipeline compile \
    --spec configs/pipelines/mambasl_cmapss_smoke.yaml \
    --output pipelines/mambasl_cmapss_smoke.yaml
```

**파라미터:**
- `--spec` — 파이프라인 스펙 YAML 경로 (필수)
- `--output` — 컴파일된 KFP YAML 출력 경로 (필수)

성공 시:
```
Pipeline compiled to pipelines/mambasl_cmapss_smoke.yaml
```

### 오버라이드와 함께 컴파일

YAML을 수정하지 않고 설정을 변경하여 컴파일:

```bash
kfp-workflow pipeline compile \
    --spec configs/pipelines/mambasl_cmapss_smoke.yaml \
    --output pipelines/experiment_large_model.yaml \
    --set model.config.d_model=128 \
    --set model.config.d_state=32 \
    --set train.max_epochs=100 \
    --set train.patience=15
```

이렇게 하면 변경된 설정이 컴파일된 YAML에 내장됩니다.

### 컴파일 결과 확인

컴파일된 YAML 파일을 열어 파이프라인 구조를 확인할 수 있습니다:

```bash
head -50 pipelines/mambasl_cmapss_smoke.yaml
```

## 파이프라인 제출

### 사전 요구사항

파이프라인 제출에는 다음이 필요합니다:
- 실행 중인 Kubernetes 클러스터
- Kubeflow Pipelines가 설치된 상태
- 클러스터에 대한 kubectl 접근 권한
- 파이프라인용 Docker 이미지가 클러스터에 로드된 상태

### 기본 제출

```bash
kfp-workflow pipeline submit \
    --spec configs/pipelines/mambasl_cmapss_smoke.yaml
```

이 명령은 내부적으로:
1. 스펙을 로드하고 검증
2. 파이프라인을 컴파일
3. `kubectl port-forward`로 KFP API에 연결
4. `kfp.Client`를 통해 파이프라인 제출
5. Run ID를 반환

성공 시:
```
Pipeline submitted. Run ID: a1b2c3d4-e5f6-7890-abcd-ef1234567890
```

### 인증 옵션

다중 사용자 환경에서 인증을 설정할 수 있습니다:

```bash
kfp-workflow pipeline submit \
    --spec configs/pipelines/mambasl_cmapss_smoke.yaml \
    --user user@example.com \
    --host "http://127.0.0.1:8888" \
    --namespace kubeflow-user-example-com
```

| 옵션 | 설명 | 기본값 |
|------|------|--------|
| `--namespace` | K8s 네임스페이스 | 스펙의 `runtime.namespace` |
| `--host` | KFP API 호스트 | 스펙의 `runtime.host` |
| `--user` | Kubeflow 사용자 ID 헤더 | `user@example.com` |
| `--existing-token` | Bearer 인증 토큰 | None |
| `--cookies` | 쿠키 헤더 | None |

## 벤치마크 워크플로우와의 관계

벤치마크도 동일하게 "컴파일 가능 + 즉시 제출 가능"한 KFP 워크플로우입니다:

```bash
kfp-workflow benchmark compile \
    --spec configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml \
    --output pipelines/mambasl_cmapss_kepler_smoke.yaml

kfp-workflow benchmark submit \
    --spec configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml
```

차이는 DAG 내용입니다. 학습 파이프라인은 `load_data → preprocess → train → evaluate → save_model`이고, 벤치마크는 "모델 배포 → Ready 대기 → 시나리오 replay + 메트릭 수집 → 정리" 흐름으로 조립됩니다. 상세 실행 예시는 [12. 벤치마크 워크플로우](12_벤치마크_워크플로우.md)를 참고하세요.

### 오버라이드와 함께 제출

컴파일과 마찬가지로 `--set`으로 설정을 변경하여 제출할 수 있습니다:

```bash
kfp-workflow pipeline submit \
    --spec configs/pipelines/mambasl_cmapss_smoke.yaml \
    --set dataset.config.fd[0].fd_name=FD003 \
    --set train.learning_rate=0.0005 \
    --set train.max_epochs=50
```

## 실전 예제: 전체 흐름

```bash
# 1. 스펙 검증
kfp-workflow spec validate --spec configs/pipelines/mambasl_cmapss_smoke.yaml

# 2. 컴파일 (선택사항 — submit이 자동으로 수행)
kfp-workflow pipeline compile \
    --spec configs/pipelines/mambasl_cmapss_smoke.yaml \
    --output pipelines/mambasl_cmapss_smoke.yaml

# 3. 제출
kfp-workflow pipeline submit \
    --spec configs/pipelines/mambasl_cmapss_smoke.yaml
```

> `pipeline submit`은 자동으로 컴파일을 수행하므로, 별도로 `pipeline compile`을 먼저 실행할 필요는 없습니다. 다만, 컴파일된 YAML을 검토하거나 버전 관리하고 싶을 때 유용합니다.

## 문제 해결

### 포트포워딩 실패
```
Error: unable to forward port because pod is not running
```
→ KFP 서비스가 실행 중인지 확인:
```bash
kubectl get pods -n kubeflow | grep ml-pipeline
```

### 이미지 풀 실패
```
ErrImagePull: kfp-workflow:latest
```
→ Docker 이미지가 클러스터에 로드되었는지 확인:
```bash
docker save kfp-workflow:latest -o /tmp/kfp-workflow-latest.tar
```

이 로컬 환경에서는 호스트에서 직접 `ctr`를 호출하지 않고, privileged helper pod + `nerdctl`로 containerd에 적재하는 검증된 흐름을 사용합니다. 전체 절차는 [09. Docker 이미지 빌드](09_Docker_이미지_빌드.md)의 "로컬 클러스터에 이미지 적재" 절을 따르세요.

### 네임스페이스 없음
→ 먼저 클러스터 부트스트랩을 실행하세요:
```bash
kfp-workflow cluster bootstrap --spec configs/pipelines/mambasl_cmapss_smoke.yaml
```
