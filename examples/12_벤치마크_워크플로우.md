# 12. 벤치마크 워크플로우

## 개요

kfp-workflow의 벤치마크는 다음 세 요소를 하나의 워크플로우로 묶습니다:

- **model** — 임시로 배포할 KServe `InferenceService`
- **scenario** — 어떤 데이터를 어떤 순서와 주기로 보낼지 정의
- **metrics** — 시나리오 실행 중 어떤 메트릭을 수집할지 정의

이 프로젝트에 포함된 대표 예시는 `configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml`입니다. 이 스펙은 다음 동작을 수행합니다:

1. `mambasl-cmapss` 모델을 임시 `InferenceService`로 배포
2. C-MAPSS FD001 일부를 1Hz 간격으로 predictor에 전송
3. Kepler가 노출한 predictor container energy를 Prometheus로 조회
4. 최종 결과를 benchmark PVC의 `results.json`으로 저장

## 아키텍처

```
BenchmarkSpec
├── model
│   └── KServe InferenceService 배포 설정
├── scenario
│   ├── dataset  → 입력 레코드/섹션 생성
│   └── pipeline → 요청 순서, 속도, 실패 처리
└── metrics
    └── 시나리오 시작/종료 전후 메트릭 수집
```

현재 기본 제공 구현:

- `cmapss-timeseries` — C-MAPSS 멀티변량 시계열 섹션 생성
- `sequential-replay` — 섹션을 하나씩 predictor에 순차 전송
- `kepler-energy` — `kepler_container_joules_total` delta 계산

## shipped smoke benchmark

### 최상위 BenchmarkSpec

```yaml
# configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml
metadata:
  name: mambasl-cmapss-benchmark-smoke
  description: MambaSL C-MAPSS benchmark smoke test with Kepler energy capture
  version: v1

runtime:
  namespace: kubeflow-user-example-com
  image: kfp-workflow:latest
  image_pull_policy: IfNotPresent
  service_account: default-editor
  host: "http://127.0.0.1:8888"
  port_forward_namespace: kubeflow
  port_forward_service: svc/ml-pipeline

storage:
  data_pvc: dataset-store
  model_pvc: model-store
  results_pvc: benchmark-store
  storage_class: local-path
  data_size: 8Gi
  model_size: 8Gi
  results_size: 8Gi
  data_mount_path: /mnt/data
  model_mount_path: /mnt/models
  results_mount_path: /mnt/benchmarks

model:
  model_name: mambasl-cmapss
  model_version: v1
  model_pvc: model-store
  model_subpath: mambasl-cmapss/v1
  predictor_image: kfp-workflow:latest
  service_name: mbsl-bench-smoke
  cleanup: true
  wait_timeout: 300

scenario:
  ref: scenarios/mambasl_cmapss_timeseries_smoke.yaml

metrics:
  - ref: metrics/kepler_predictor_energy.yaml
```

핵심 포인트:

- `storage.results_pvc`가 벤치마크 결과 저장소입니다.
- `model` 블록은 벤치마크 동안만 사용할 임시 `InferenceService`를 정의합니다.
- `scenario`와 `metrics`는 별도 YAML로 분리해 재사용할 수 있습니다.

### scenario ref

```yaml
# configs/benchmarks/scenarios/mambasl_cmapss_timeseries_smoke.yaml
kind: inline
dataset:
  kind: cmapss-timeseries
  config:
    fd:
      - fd_name: FD001
        unit_ids: [1, 2, 3]
        max_sections: 5
    feature_mode: settings_plus_sensors
    norm_mode: condition_minmax
pipeline:
  kind: sequential-replay
  config:
    interval_hz: 1.0
    request_timeout: 30
```

이 시나리오는 FD001에서 일부 유닛만 읽고, 최대 5개 섹션만 1초 간격으로 전송하는 작은 smoke benchmark입니다.

### metric ref

```yaml
# configs/benchmarks/metrics/kepler_predictor_energy.yaml
kind: kepler-energy
config:
  prometheus_url: http://kube-prometheus-stack-prometheus.monitoring.svc.cluster.local:9090
  metric_name: kepler_container_joules_total
  container_name: kserve-container
  mode: dynamic
  settle_seconds: 20
  poll_interval_seconds: 1
```

`settle_seconds: 20`은 이 로컬 클러스터에서 Kepler counter 반영이 지연되는 특성을 반영한 값입니다. 더 짧게 줄이면 마지막 요청이 energy delta에 반영되지 않을 수 있습니다.

## YAML과 Python 확장

scenario와 metrics는 YAML ref뿐 아니라 Python 심볼로도 정의할 수 있습니다.

```yaml
scenario:
  ref: benchmark_defs/my_scenario.py:SCENARIO

metrics:
  - ref: benchmark_defs/my_metric.py:METRIC
```

또는 dataset / pipeline / metric 노드에서 `kind: python`과 `ref:`를 직접 둘 수 있습니다. `spec validate --type benchmark`는 ref를 실제로 해석하고 표준 인터페이스를 검사합니다.

## 실행 순서

### 1. 스펙 검증

```bash
kfp-workflow spec validate \
    --spec configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml \
    --type benchmark
```

성공하면 materialized benchmark payload가 출력됩니다. 여기서 `scenario`와 `metrics` ref가 실제로 해석되었는지 먼저 확인하는 것이 좋습니다.

### 2. benchmark 전용 PVC 부트스트랩

```bash
kfp-workflow cluster bootstrap \
    --spec configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml \
    --type benchmark
```

이 명령은 최소 세 PVC를 준비합니다:

- `dataset-store`
- `model-store`
- `benchmark-store`

미리보기만 하려면:

```bash
kfp-workflow cluster bootstrap \
    --spec configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml \
    --type benchmark \
    --dry-run
```

### 3. 컴파일

```bash
kfp-workflow benchmark compile \
    --spec configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml \
    --output pipelines/mambasl_cmapss_kepler_smoke.yaml
```

학습 파이프라인과 달리 benchmark DAG는 side effect가 크므로, 이 프로젝트의 benchmark task들은 KFP cache를 사용하지 않도록 컴파일됩니다.

### 4. 제출

```bash
kfp-workflow benchmark submit \
    --spec configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml
```

필요하면 `--namespace`, `--host`, `--user`, `--existing-token`, `--cookies`, `--set`을 그대로 사용할 수 있습니다.

예시:

```bash
kfp-workflow benchmark submit \
    --spec configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml \
    --set model.service_name=mbsl-bench-smoke-alt
```

### 5. 실행 대기 및 상태 확인

```bash
kfp-workflow pipeline run wait <run_id>
kfp-workflow pipeline run get <run_id>
kfp-workflow benchmark get <run_id>
kfp-workflow benchmark download <run_id>
```

벤치마크 실행도 KFP run이므로 모니터링은 기존 `pipeline run *` 명령을 그대로 사용합니다.

## 결과 확인

### 결과 파일 위치

benchmark runtime은 다음 경로 패턴으로 결과를 저장합니다:

```text
/mnt/benchmarks/benchmark-results/<benchmark-name>/<timestamp>-<benchmark-pod-hostname>/results.json
```

정확한 최종 경로는 `results.json` 안의 `results_path` 필드를 기준으로 확인하는 것이 가장 안전합니다.

실제 payload에는 최소 다음 필드가 들어 있습니다:

```json
{
  "benchmark_name": "mambasl-cmapss-benchmark-smoke",
  "status": "succeeded",
  "target": { "...": "..." },
  "scenario": {
    "request_count": 5
  },
  "metrics": {
    "metric_0": {
      "delta_joules": 27.963000000000008
    }
  }
}
```

### PVC에서 결과 읽기

결과를 직접 확인하려면 benchmark PVC를 잠깐 마운트한 helper pod를 사용할 수 있습니다:

```bash
kubectl apply -f - <<'YAML'
apiVersion: v1
kind: Pod
metadata:
  name: benchmark-results-viewer
  namespace: kubeflow-user-example-com
spec:
  restartPolicy: Never
  containers:
  - name: shell
    image: busybox:1.36
    command: ["sh", "-c", "sleep 3600"]
    volumeMounts:
    - name: bench
      mountPath: /mnt/benchmarks
  volumes:
  - name: bench
    persistentVolumeClaim:
      claimName: benchmark-store
YAML
kubectl wait --for=condition=Ready pod/benchmark-results-viewer -n kubeflow-user-example-com --timeout=120s
kubectl exec -n kubeflow-user-example-com benchmark-results-viewer -- \
  sh -lc 'find /mnt/benchmarks -name results.json | sort'
kubectl delete pod -n kubeflow-user-example-com benchmark-results-viewer --ignore-not-found=true
```

특정 run 디렉터리를 알면 `cat`으로 내용을 바로 확인할 수 있습니다.

- `metrics.metric_0.delta_joules > 0`

## 문제 해결

### benchmark-store PVC가 없음

학습용 bootstrap이 아니라 benchmark 타입으로 실행해야 합니다:

```bash
kfp-workflow cluster bootstrap \
    --spec configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml \
    --type benchmark
```

### 마지막 energy delta가 0에 가까움

이 로컬 클러스터에서는 Kepler metric 반영이 지연됩니다. `configs/benchmarks/metrics/kepler_predictor_energy.yaml`의 `settle_seconds`를 너무 짧게 줄이지 마세요.

### predictor 이미지 pull 실패

학습 파이프라인과 동일하게, 로컬 클러스터에서는 helper pod + `nerdctl` import 흐름을 사용합니다. 절차는 [09. Docker 이미지 빌드](09_Docker_이미지_빌드.md)를 따르세요.
