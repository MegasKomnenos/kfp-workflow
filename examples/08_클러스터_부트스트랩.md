# 08. 클러스터 부트스트랩

## 개요

파이프라인이나 벤치마크를 실행하기 전에 Kubernetes 클러스터에 필요한 스토리지 리소스(PVC)를 준비해야 합니다. `cluster bootstrap` 명령어는 스펙의 `storage` 설정을 읽어 PVC 매니페스트를 자동 생성하고 적용합니다.

### 부트스트랩이 생성하는 리소스

학습 파이프라인은 보통 두 개의 PVC를, 벤치마크는 세 번째 결과 PVC를 추가로 생성합니다:

| PVC | 용도 | 스펙 필드 |
|-----|------|-----------|
| **데이터 PVC** | 학습 데이터 저장 | `storage.data_pvc`, `storage.data_size` |
| **모델 PVC** | 모델 아티팩트 저장 | `storage.model_pvc`, `storage.model_size` |
| **결과 PVC** | benchmark `results.json` 저장 | `storage.results_pvc`, `storage.results_size` |

## 스토리지 설정 (스펙 파일)

```yaml
# configs/pipelines/mambasl_cmapss_smoke.yaml (발췌)
storage:
  data_pvc: workspace-data-pvc       # 데이터 PVC 이름
  data_size: 10Gi                    # 데이터 PVC 크기
  data_mount_path: /mnt/data         # 컨테이너 마운트 경로

  model_pvc: workspace-model-pvc     # 모델 PVC 이름
  model_size: 10Gi                   # 모델 PVC 크기
  model_mount_path: /mnt/models      # 컨테이너 마운트 경로

  storage_class: nfs-client          # StorageClass 이름

runtime:
  namespace: kubeflow-user-example-com  # PVC가 생성될 네임스페이스
```

벤치마크 스펙에서는 결과 PVC가 추가됩니다:

```yaml
# configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml (발췌)
storage:
  data_pvc: dataset-store
  model_pvc: model-store
  results_pvc: benchmark-store
  data_size: 8Gi
  model_size: 8Gi
  results_size: 8Gi
  results_mount_path: /mnt/benchmarks
```

## 부트스트랩 명령어

### Dry Run (미리보기)

실제로 적용하지 않고 생성될 매니페스트만 확인합니다:

```bash
kfp-workflow cluster bootstrap \
    --spec configs/pipelines/mambasl_cmapss_smoke.yaml \
    --dry-run
```

출력:
```json
[
  {
    "apiVersion": "v1",
    "kind": "PersistentVolumeClaim",
    "metadata": {
      "name": "workspace-data-pvc",
      "namespace": "kubeflow-user-example-com"
    },
    "spec": {
      "accessModes": ["ReadWriteOnce"],
      "storageClassName": "nfs-client",
      "resources": {
        "requests": {
          "storage": "10Gi"
        }
      }
    }
  },
  {
    "apiVersion": "v1",
    "kind": "PersistentVolumeClaim",
    "metadata": {
      "name": "workspace-model-pvc",
      "namespace": "kubeflow-user-example-com"
    },
    "spec": {
      "accessModes": ["ReadWriteOnce"],
      "storageClassName": "nfs-client",
      "resources": {
        "requests": {
          "storage": "10Gi"
        }
      }
    }
  }
]
Dry run — manifests printed above, nothing applied.
```

벤치마크 스펙 미리보기:

```bash
kfp-workflow cluster bootstrap \
    --spec configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml \
    --type benchmark \
    --dry-run
```

이 경우 출력 JSON에는 `dataset-store`, `model-store`, `benchmark-store` 세 PVC가 포함됩니다.

### 실제 적용

```bash
kfp-workflow cluster bootstrap \
    --spec configs/pipelines/mambasl_cmapss_smoke.yaml
```

출력:
```json
[...]
Applied PersistentVolumeClaim 'workspace-data-pvc'
Applied PersistentVolumeClaim 'workspace-model-pvc'
Cluster bootstrap complete.
```

명령어가 수행하는 작업:

1. 네임스페이스가 없으면 생성 (이미 있으면 무시)
2. 데이터 PVC를 `kubectl apply`로 적용 (멱등)
3. 모델 PVC를 `kubectl apply`로 적용 (멱등)
4. 벤치마크 스펙인 경우 결과 PVC도 `kubectl apply`로 적용 (멱등)

> `kubectl apply`를 사용하므로 이미 존재하는 PVC에 대해 안전하게 재실행할 수 있습니다 (멱등성 보장).

## 부트스트랩 워크플로

### 초기 설정 순서

```
1. 파이프라인 스펙 작성/검증
   └─ kfp-workflow spec validate --spec ...

2. 클러스터 부트스트랩 (PVC 생성)
   └─ kfp-workflow cluster bootstrap --spec ...

3. PVC 상태 확인
   └─ kubectl get pvc -n kubeflow-user-example-com

4. 파이프라인 제출
   └─ kfp-workflow pipeline submit --spec ...

5. 또는 벤치마크 제출
   └─ kfp-workflow benchmark submit --spec ...
```

### PVC 상태 확인

부트스트랩 후 PVC가 정상적으로 바인딩되었는지 확인합니다:

```bash
kubectl get pvc -n kubeflow-user-example-com
```

정상 출력:
```
NAME                   STATUS   VOLUME              CAPACITY   ACCESS MODES   STORAGECLASS   AGE
workspace-data-pvc     Bound    pv-abc123           10Gi       RWO            nfs-client     5m
workspace-model-pvc    Bound    pv-def456           10Gi       RWO            nfs-client     5m
```

벤치마크 스펙을 부트스트랩했다면 `benchmark-store`도 함께 보여야 합니다.

**STATUS 해석:**
- `Bound`: 정상 — PV에 바인딩 완료
- `Pending`: 대기 — StorageClass가 프로비저닝 중이거나 문제 있음
- `Lost`: 문제 — 바인딩된 PV가 사라짐

## 문제 해결

### PVC가 Pending 상태

```bash
# PVC 이벤트 확인
kubectl describe pvc workspace-data-pvc -n kubeflow-user-example-com
```

**일반적인 원인:**
- StorageClass가 존재하지 않음
- 스토리지 프로비저너(NFS 등)가 정상 동작하지 않음
- 스토리지 용량 부족

```bash
# StorageClass 확인
kubectl get storageclass

# 프로비저너 Pod 확인 (NFS 예시)
kubectl get pods -n default | grep nfs
```

### kubectl 권한 부족

```bash
# 현재 사용자 권한 확인
kubectl auth can-i create pvc -n kubeflow-user-example-com
```

### 네임스페이스가 없는 경우

`cluster bootstrap`은 네임스페이스를 자동으로 생성하려 시도합니다. 수동 생성이 필요한 경우:

```bash
kubectl create namespace kubeflow-user-example-com
```

### 벤치마크 결과 PVC가 없는 경우

`benchmark submit` 전에 아래 명령으로 benchmark 타입으로 부트스트랩했는지 확인합니다:

```bash
kfp-workflow cluster bootstrap \
    --spec configs/benchmarks/mambasl_cmapss_kepler_smoke.yaml \
    --type benchmark
```
