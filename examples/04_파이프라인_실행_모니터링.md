# 04. 파이프라인 실행 모니터링

## 개요

파이프라인을 제출한 후에는 실행 상태를 확인하고, 완료를 대기하고, 로그를 확인하고, 필요시 실행을 중단할 수 있습니다. `kfp-workflow pipeline run` 명령어 그룹이 이 모든 기능을 제공합니다.

## 실행 모니터링 명령어

### 실행 목록 조회 (`run list`)

제출된 파이프라인 실행들을 목록으로 조회합니다:

```bash
kfp-workflow pipeline run list
```

출력 예시:
```
┌──────────────────────────────────────────────────────────────────────────┐
│                           Pipeline Runs                                 │
├──────────┬──────────────────────┬───────────┬───────────┬──────────────┤
│ ID       │ NAME                 │ STATE     │ CREATED   │ FINISHED     │
├──────────┼──────────────────────┼───────────┼───────────┼──────────────┤
│ a1b2c3d4 │ mambasl-cmapss-smoke │ SUCCEEDED │ 2026-03-… │ 2026-03-…   │
│ e5f6a7b8 │ mambasl-cmapss-fd003 │ RUNNING   │ 2026-03-… │             │
└──────────┴──────────────────────┴───────────┴───────────┴──────────────┘
```

**옵션:**

| 옵션 | 설명 | 기본값 |
|------|------|--------|
| `--namespace` | K8s 네임스페이스 | `kubeflow-user-example-com` |
| `--experiment-id` | 특정 실험 ID로 필터링 | None |
| `--page-size` | 반환할 실행 수 | 20 |
| `--sort-by` | 정렬 기준 | `created_at desc` |
| `--host` | KFP API 호스트 | `http://127.0.0.1:8888` |
| `--user` | Kubeflow 사용자 ID | `user@example.com` |

#### 실험별 필터링

```bash
kfp-workflow pipeline run list --experiment-id "abc12345-..."
```

#### 정렬 변경

```bash
# 이름 오름차순
kfp-workflow pipeline run list --sort-by "display_name asc"

# 최근 생성순 (기본)
kfp-workflow pipeline run list --sort-by "created_at desc"
```

### 실행 상세 조회 (`run get`)

특정 실행의 상세 정보를 확인합니다:

```bash
kfp-workflow pipeline run get <RUN_ID>
```

출력 예시:
```
Run ID      : a1b2c3d4-e5f6-7890-abcd-ef1234567890
Name        : mambasl-cmapss-smoke
State       : SUCCEEDED
Created     : 2026-03-23 10:30:00+00:00
Finished    : 2026-03-23 10:45:00+00:00
Experiment  : Default
Workflow    : mambasl-cmapss-smoke-abc123
WF Phase    : Succeeded
WF Progress : 5/5
```

Argo Workflow 객체를 찾을 수 있으면 `run get`은 workflow 이름, phase, progress, workflow message, pending/failed 노드 정보까지 함께 보여줍니다.

**상태 색상:**
- 초록: `SUCCEEDED` — 성공
- 빨강: `FAILED` — 실패
- 파랑: `RUNNING` — 실행 중
- 노랑: `PENDING` — 대기 중
- 회색: `CANCELED` — 취소됨

**옵션:**

| 옵션 | 설명 | 기본값 |
|------|------|--------|
| `--namespace` | K8s 네임스페이스 | `kubeflow-user-example-com` |
| `--host` | KFP API 호스트 | `http://127.0.0.1:8888` |
| `--user` | Kubeflow 사용자 ID | `user@example.com` |

### 실행 완료 대기 (`run wait`)

실행이 종료 상태에 도달할 때까지 블로킹 대기합니다:

```bash
kfp-workflow pipeline run wait <RUN_ID>
```

출력 예시:
```
⠋ Waiting for run a1b2c3d4...
Run a1b2c3d4 finished: SUCCEEDED
Workflow: mambasl-cmapss-smoke-abc123 (Succeeded, 5/5)
```

실패 시:
```
Run a1b2c3d4 finished: FAILED
Error: Component 'train' failed: OutOfMemory
```

실패(`FAILED`) 또는 취소(`CANCELED`) 시 exit code 1을 반환하므로, 스크립트에서 활용할 수 있습니다.
타임아웃 시에는 가능한 경우 Argo Workflow의 pending/failed 노드와 workflow message를 함께 출력합니다.

**옵션:**

| 옵션 | 설명 | 기본값 |
|------|------|--------|
| `--timeout` | 타임아웃 (초) | 3600 (1시간) |
| `--namespace` | K8s 네임스페이스 | `kubeflow-user-example-com` |
| `--host` | KFP API 호스트 | `http://127.0.0.1:8888` |

```bash
# 30분 타임아웃으로 대기
kfp-workflow pipeline run wait <RUN_ID> --timeout 1800
```

### 로그 확인 (`run logs`)

파이프라인 실행의 각 컴포넌트 Pod 로그를 확인합니다:

```bash
kfp-workflow pipeline run logs <RUN_ID>
```

출력 예시:
```
─── load-data-system-container-impl-abc123 ───
Loading C-MAPSS FD001 dataset...
Downloaded 6.2MB to /mnt/data/cmapss/
Train samples: 20631, Test samples: 100

─── preprocess-system-container-impl-def456 ───
Feature mode: settings_plus_sensors
Window size: 30, Max RUL: 125.0
Train: 18524 windows, Val: 2107 windows, Test: 100 windows

─── train-system-container-impl-ghi789 ───
Epoch 1/2: train_rmse=45.32, val_rmse=38.21
Epoch 2/2: train_rmse=32.18, val_rmse=29.45
Best epoch: 2 (val_rmse=29.45)
```

#### 특정 스텝 로그만 확인

```bash
# train 스텝만 필터링
kfp-workflow pipeline run logs <RUN_ID> --step train

# evaluate 스텝만 필터링
kfp-workflow pipeline run logs <RUN_ID> --step evaluate
```

**옵션:**

| 옵션 | 설명 | 기본값 |
|------|------|--------|
| `--step` | 스텝 이름 부분 문자열로 필터링 | None (전체) |
| `--namespace` | K8s 네임스페이스 | `kubeflow-user-example-com` |

> `run logs`는 Kubernetes API를 직접 사용하므로 `kubectl` 접근 권한이 필요합니다. KFP API가 아닌 K8s Pod 로그를 직접 조회합니다.
> impl Pod가 이미 정리된 경우 driver Pod 로그로 fallback하고, 그것도 없으면 workflow 진단 정보를 출력합니다.

### 실행 중단 (`run terminate`)

실행 중인 파이프라인을 중단(취소)합니다:

```bash
kfp-workflow pipeline run terminate <RUN_ID>
```

출력:
```
Run a1b2c3d4 terminated.
```

이 명령은 KFP API를 통해 실행을 취소 요청합니다. 이미 실행 중인 Pod가 즉시 종료되지는 않을 수 있습니다.

## 실험 관리

### 실험 목록 조회 (`experiment list`)

```bash
kfp-workflow pipeline experiment list
```

출력 예시:

## JSON 출력 모드

모든 조회 명령에 `--json` 플래그를 추가하여 스크립팅에 적합한 JSON 출력을 받을 수 있습니다:

```bash
# 실행 목록을 JSON으로
kfp-workflow --json pipeline run list
```

출력:
```json
[
  {
    "run_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "display_name": "mambasl-cmapss-smoke",
    "state": "SUCCEEDED",
    "created_at": "2026-03-23 10:30:00+00:00",
    "finished_at": "2026-03-23 10:45:00+00:00"
  }
]
```

```bash
# 특정 실행을 JSON으로
kfp-workflow --json pipeline run get <RUN_ID>
```

출력:
```json
{
  "run_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "display_name": "mambasl-cmapss-smoke",
  "state": "SUCCEEDED",
  "created_at": "2026-03-23 10:30:00+00:00",
  "finished_at": "2026-03-23 10:45:00+00:00",
  "experiment_id": "abc12345-...",
  "error": null,
  "workflow_name": "mambasl-cmapss-smoke-abc123",
  "workflow_phase": "Succeeded",
  "workflow_progress": "5/5",
  "workflow_message": "",
  "pending_nodes": [],
  "failed_nodes": []
}
```

> `--json` 플래그는 최상위 `kfp-workflow` 명령어 바로 뒤에 위치해야 합니다 (서브커맨드 앞).

## 실전 예제: 제출부터 결과 확인까지

```bash
# 1. 파이프라인 제출
kfp-workflow pipeline submit \
    --spec configs/pipelines/mambasl_cmapss_smoke.yaml

# 출력: Pipeline submitted. Run ID: a1b2c3d4-...

# 2. 완료 대기
kfp-workflow pipeline run wait a1b2c3d4-e5f6-7890-abcd-ef1234567890

# 3. 결과 확인
kfp-workflow pipeline run get a1b2c3d4-e5f6-7890-abcd-ef1234567890

# 4. 로그 확인 (평가 결과만)
kfp-workflow pipeline run logs a1b2c3d4-e5f6-7890-abcd-ef1234567890 --step evaluate
```

### 스크립트 자동화 예제

```bash
#!/bin/bash
set -e

# 제출 후 Run ID 추출
RUN_ID=$(kfp-workflow --json pipeline submit \
    --spec configs/pipelines/mambasl_cmapss_smoke.yaml \
    | python3 -c "import sys,json; print(json.load(sys.stdin)['run_id'])" 2>/dev/null \
    || kfp-workflow pipeline submit \
        --spec configs/pipelines/mambasl_cmapss_smoke.yaml \
        2>&1 | grep -oP 'Run ID: \K.*')

echo "Submitted: $RUN_ID"

# 완료 대기 (1시간 타임아웃)
kfp-workflow pipeline run wait "$RUN_ID" --timeout 3600

# 결과를 JSON으로 저장
kfp-workflow --json pipeline run get "$RUN_ID" > "results/${RUN_ID}.json"

echo "Results saved to results/${RUN_ID}.json"
```

## 문제 해결

### 포트포워딩 관련

`run get`, `run list`, `run wait`, `run terminate`는 KFP API를 사용하므로 포트포워딩이 필요합니다. 연결 실패 시:

```bash
# 수동 포트포워딩
kubectl port-forward -n kubeflow svc/ml-pipeline 8888:8888
```

### 로그 조회 실패

`run logs`는 Kubernetes API를 직접 사용합니다:

```bash
# kubectl 접근 확인
kubectl get pods -n kubeflow-user-example-com

# 특정 Run ID의 Pod 직접 확인
kubectl get pods -n kubeflow-user-example-com -l pipeline/runid=<RUN_ID>
```

### Pod를 찾을 수 없음

```
No matching pods found.
```
→ impl Pod가 아직 생성되지 않았거나 이미 삭제된 상태일 수 있습니다. 최신 `run get` / `run wait` 출력에서 workflow 상태를 먼저 확인하고, 필요하면 driver Pod 또는 Workflow 진단 정보를 사용하세요.
