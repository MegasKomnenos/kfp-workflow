# 00. 프로젝트 개요

## 한 줄 설명

`kfp-workflow`는 Kubeflow Pipelines 학습, KServe 서빙, Katib 기반 튜닝, 임시 서빙 기반 벤치마크, 레지스트리, 스토리지 부트스트랩을 하나의 루트 CLI로 운영하는 프로젝트입니다.

## 현재 지원 범위

루트 플러그인으로 바로 사용할 수 있는 모델은 다음 세 가지입니다.

- `mambasl-cmapss`
- `mrhysp-cmapss`
- `softs-cmapss`

`models/timemixer-new`는 저장소 안에 존재하지만, 현재 루트 플러그인 레지스트리에 연결되어 있지 않습니다.

## 구조

```text
kfp-workflow CLI domains
├── pipeline    학습 파이프라인 컴파일/제출/조회
├── benchmark   벤치마크 DAG 구성과 결과 수집
├── serve       KServe InferenceService 관리
├── tune        Katib 탐색 공간 해석, 실험 생성, 결과 조회
├── registry    모델/데이터셋 레지스트리
└── cluster     스토리지 PVC 부트스트랩

src/kfp_workflow/ modules
├── pipeline/   학습 파이프라인 컴파일/제출 로직
├── benchmark/  벤치마크 compile/runtime/history
├── serving/    KServe 연동과 predictor
├── tune/       search space, Katib manifest, 결과 처리
└── plugins/    루트 통합 모델 어댑터
```

## 주요 개념

### 스펙 중심 운영

루트 워크플로우는 스펙 중심으로 구동됩니다.

- `configs/pipelines/`: 학습용 YAML
- `configs/serving/`: 서빙용 YAML
- `configs/tuning/`: 튜닝용 YAML
- `configs/benchmarks/`: 벤치마크 예제 입력. 현재 벤치마크 명령은 YAML과 Python 정의를 모두 받을 수 있습니다.

### 학습 DAG

```text
load_data -> preprocess -> train -> evaluate -> save_model
```

### 벤치마크 모델

벤치마크는 세 요소를 묶습니다.

- `model`: 임시 KServe 배포 대상
- `scenario`: 어떤 입력을 어떤 방식으로 재생할지
- `metrics`: 실행 중 수집할 메트릭

### 튜닝 모델

프로젝트가 HPO 오케스트레이션을 소유하고, 루트 공개 UX는 Katib experiment 제출/조회 흐름입니다.

사용자 관점의 기본 흐름은 `kfp-workflow tune submit --spec ...` 입니다.
