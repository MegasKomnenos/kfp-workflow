# 00. 프로젝트 개요

## kfp-workflow란?

kfp-workflow는 Kubeflow Pipelines v2 기반의 **ML 학습 파이프라인**, 임시 KServe 배포 기반의 **벤치마크 워크플로우**, 그리고 KServe 기반의 **추론 서빙**을 하나의 CLI로 관리하는 워크플로우 매니저입니다.

YAML 스펙 파일 하나로 학습부터 배포까지 전체 ML 라이프사이클을 선언적으로 정의하고 실행할 수 있습니다.

## 핵심 철학

### 선언형 워크플로우
모든 파이프라인 동작은 YAML 스펙 파일로 정의됩니다. 코드를 수정하지 않고 YAML만 변경하여 다양한 실험을 실행할 수 있습니다.

### 단일 CLI 진입점
`kfp-workflow` 하나의 명령어로 검증, 컴파일, 제출, 모니터링, 서빙, 레지스트리 관리까지 모든 작업을 수행합니다.

### 재현 가능한 실험
컴파일된 파이프라인 YAML에는 모든 설정이 포함되어 있어, 동일한 파일로 언제든 동일한 실험을 재현할 수 있습니다.

### 모듈형 모델 플러그인
모델별 ML 로직은 플러그인으로 분리되어 있습니다. 새로운 모델을 추가할 때 파이프라인 오케스트레이션 코드를 건드릴 필요가 없습니다.

## 아키텍처 개요

프로젝트는 오케스트레이션 계층과 모델 패키지 계층으로 구성되며, 오케스트레이션 계층 안에는 학습/벤치마크/서빙 기능이 함께 포함됩니다:

```
┌──────────────────────────────────────────────────────┐
│                   kfp-workflow                        │
│  (오케스트레이션: CLI, 학습, 벤치마크, 서빙, 레지스트리)  │
│                                                      │
│  src/kfp_workflow/                                   │
│  ├── cli/          → Typer CLI + Rich 출력           │
│  ├── pipeline/     → KFP 학습 파이프라인 컴파일/제출  │
│  ├── benchmark/    → 벤치마크 DAG + 시나리오 실행     │
│  ├── components/   → 5개 파이프라인 컴포넌트          │
│  ├── plugins/      → ModelPlugin ABC + 어댑터        │
│  ├── tune/         → HPO 엔진 (Optuna) + Katib 빌더  │
│  ├── serving/      → KServe CRUD + 커스텀 예측기      │
│  └── registry/     → 모델/데이터셋 파일 레지스트리     │
└────────────────────────┬─────────────────────────────┘
                         │ import
┌────────────────────────▼─────────────────────────────┐
│                 models/*-new/                        │
│  (ML 로직: 데이터, 전처리, 모델, 학습, 평가)           │
│                                                      │
│  예: mambasl-new / multirocket-new / softs-new       │
│  ├── cmapss/       → C-MAPSS 데이터 + 학습 로직       │
│  ├── model.py      → 모델/변환기 구현                │
│  └── kubeflow/     → Katib/HPO 보조 코드             │
└──────────────────────────────────────────────────────┘
```

**kfp-workflow**는 "무엇을 실행할지"를 관리하고, **모델 패키지**는 "어떻게 학습할지"의 ML 로직을 담당합니다.

## 핵심 기능

### 1. 학습 파이프라인 DAG

```
load_data → preprocess → train → evaluate → save_model
```

5개의 KFP 컴포넌트가 순차적으로 실행됩니다. 각 컴포넌트는:
- 전체 `PipelineSpec`을 JSON으로 전달받아 자체 완결적으로 동작
- `ModelPlugin`에 작업을 위임하여 모델별 로직 실행
- 무거운 데이터(numpy 배열)는 PVC의 `.npy` 파일로 저장하고, 경로만 다음 스테이지로 전달

### 2. 모듈형 플러그인 시스템

`ModelPlugin` 추상 클래스를 구현하여 새로운 모델을 추가할 수 있습니다:

```python
class ModelPlugin(ABC):
    def name() -> str           # 플러그인 식별자
    def load_data(...)          # 데이터 로드
    def preprocess(...)         # 전처리
    def train(...)              # 학습
    def evaluate(...)           # 평가
    def save_model(...)         # 모델 저장
    def predict(...)            # 추론 (서빙용)
```

플러그인은 선택적으로 HPO 메서드도 구현할 수 있습니다:
- `hpo_search_space()` — 내장 탐색 공간 반환
- `hpo_base_config()` — HPO 기본 설정 반환
- `hpo_objective()` — 단일 시행 실행

현재 등록된 플러그인:
- `mambasl-cmapss` — MambaSL 상태공간 모델
- `mrhysp-cmapss` — MR-HY-SP / MultiRocket 기반 모델
- `softs-cmapss` — SOFTS 기반 모델

### 3. Helm 스타일 `--set` 오버라이드

YAML 파일을 수정하지 않고 CLI에서 직접 설정을 변경할 수 있습니다:

```bash
kfp-workflow pipeline compile \
    --spec configs/pipelines/mambasl_cmapss_smoke.yaml \
    --output pipelines/experiment.yaml \
    --set train.max_epochs=100 \
    --set model.config.d_model=128
```

우선순위: `CLI --set` > `YAML 스펙` > `플러그인 기본값`

### 4. 하이퍼파라미터 최적화 (HPO)

프로젝트가 HPO 오케스트레이션을 소유하고, 플러그인은 탐색 공간과 목적 함수만 제공합니다:

- **로컬 HPO** — Optuna 기반 (TPE, Random, Grid 알고리즘)
- **분산 HPO** — Kubeflow Katib CRD 매니페스트 생성
- **내장 프로파일** — 플러그인이 `default`/`aggressive` 탐색 공간 제공
- **커스텀 탐색 공간** — YAML로 직접 정의 가능

```bash
kfp-workflow tune run --spec configs/tuning/mambasl_cmapss_tune.yaml \
    --set hpo.max_trials=20 --data-mount-path ./data
```

### 5. KServe 추론 서빙

학습된 모델을 KServe InferenceService로 배포:
- **Custom runtime** — 플러그인이 선택한 모델 파일을 로드하고 선택적으로 warmup 후 예측하는 커스텀 예측기
- **Standard runtime** — KServe TorchServe 등 표준 런타임

### 6. 벤치마크 워크플로우

벤치마크는 다음 세 요소의 조합입니다:
- **model** — 임시로 배포할 KServe `InferenceService`
- **scenario** — 데이터셋 + 요청 파이프라인
- **metrics** — 시나리오 실행 중 수집할 메트릭

기본 제공되는 구성 요소:

| 종류 | 키워드 | 용도 |
|------|--------|------|
| 데이터셋 소스 | `cmapss-timeseries` | 슬라이딩 윈도우 전체 (에너지/지연 측정) |
| 데이터셋 소스 | `cmapss-test-set` | 유닛당 마지막 윈도우 1개 (정확도 평가) |
| 파이프라인 | `sequential-replay` | 설정 속도(Hz)로 순차 전송 |
| 파이프라인 | `test-eval` | 속도 제한 없이 1회 전송 |
| 메트릭 | `kepler-energy` | Prometheus/Kepler container energy delta |
| 메트릭 | `cmapss-test` | RUL 임계값 기반 F1/precision/recall/accuracy |

제공되는 벤치마크:
- `mambasl_cmapss_kepler_smoke.yaml` — MambaSL FD001 일부를 1Hz로 재생, Kepler energy 수집
- `mambasl_cmapss_test.yaml` — MambaSL FD001 전체 test set 정확도 평가 (F1/precision/recall/accuracy)

### 7. 파일 기반 레지스트리

- **모델 레지스트리** — PVC의 `.model_registry.json`에 모델 메타데이터 저장
- **데이터셋 레지스트리** — PVC의 `.dataset_registry.json`에 데이터셋 정보 저장
- **런타임 통합** — `storage.data_subpath`가 비어 있으면 같은 data PVC 안의 레지스트리에서 `dataset.name` / `dataset.version` 경로 조회

### 8. 구조화된 CLI 출력

Rich 라이브러리 기반의 컬러 테이블, 상태 표시, JSON 출력 모드 지원:
- 파이프라인 실행 상태에 색상 표시 (초록=성공, 빨강=실패, 파랑=실행중)
- `--json` 플래그로 스크립팅용 JSON 출력

## 전체 워크플로우

```
1. spec validate     → 스펙 파일 검증
2. cluster bootstrap → PVC 스토리지 생성
3. tune run/katib    → (선택) 하이퍼파라미터 최적화
4. pipeline compile  → KFP YAML 컴파일
5. pipeline submit   → Kubeflow에 학습 워크플로우 제출
6. serve create      → 상시 추론 서비스 배포
7. benchmark submit  → 벤치마크 워크플로우 제출
8. pipeline run *    → 학습/벤치마크 실행 모니터링
```

```
test/
├── configs/                  # YAML 스펙 파일
│   ├── pipelines/           # 학습 파이프라인 스펙
│   ├── benchmarks/          # 벤치마크 스펙, 시나리오 ref, 메트릭 ref
│   ├── tuning/              # HPO 튜닝 스펙
│   └── serving/             # 서빙 스펙
├── docker/                  # Dockerfile
├── models/                  # 모델 패키지 (플러그인용 ML 로직)
│   └── mambasl-new/         # MambaSL C-MAPSS 모델
├── src/kfp_workflow/        # 메인 패키지
│   ├── cli/                 # CLI 명령어
│   ├── components/          # KFP 파이프라인 컴포넌트
│   ├── benchmark/           # 벤치마크 컴파일/실행/메트릭 수집
│   ├── pipeline/            # 컴파일러 + 클라이언트
│   ├── plugins/             # 모델 플러그인 시스템
│   ├── tune/                # HPO 엔진 + Katib 빌더
│   ├── registry/            # 모델/데이터셋 레지스트리
│   └── serving/             # KServe 관리
├── tests/                   # 테스트 스위트
└── examples/                # 이 튜토리얼 디렉토리
```

## 기술 스택

| 구성 요소 | 기술 | 버전 |
|-----------|------|------|
| 프레임워크 | Kubeflow Pipelines | v2 (SDK 2.15.0) |
| 서빙 | KServe | v1beta1 |
| ML 프레임워크 | PyTorch | 2.4.1 |
| 컨테이너 | Docker | CUDA 12.1 기반 |
| CLI | Typer + Rich | - |
| HPO | Optuna | >=4.5 |
| 검증 | Pydantic | v2 |
| 쿠버네티스 | kubectl + Python client | - |
