# 06. 서빙 배포 및 추론

## 생성

```bash
kfp-workflow serve create \
  --spec configs/serving/mambasl_cmapss_serve.yaml \
  --wait
```

매니페스트만 확인:

```bash
kfp-workflow serve create \
  --spec configs/serving/mambasl_cmapss_serve.yaml \
  --dry-run
```

## 조회와 삭제

```bash
kfp-workflow serve list
kfp-workflow serve get mambasl-cmapss-serving
kfp-workflow serve delete mambasl-cmapss-serving
```

## 서빙 스펙에서 중요한 값

- `metadata.name`
- `namespace`
- `model_pvc`
- `model_subpath`
- `runtime`
- `predictor_image`
- `model_name`

`model_subpath`는 단일 파일이 아니라 모델 디렉토리를 가리켜야 합니다.

## 추론 요청 예시

클러스터 내부 서비스 URL 또는 포트포워딩된 endpoint에 JSON 요청을 보냅니다.

```bash
SERVICE_NAME=mambasl-cmapss-serving

curl -X POST "http://localhost:8080/v1/models/${SERVICE_NAME}:predict" \
  -H "Content-Type: application/json" \
  -d '{"instances": [[[0.1, 0.2], [0.3, 0.4]]]}'
```

실제 입력 shape은 모델이 기대하는 시계열 shape와 일치해야 합니다.

## 참고

- `serve create`는 `--wait`와 `--timeout`을 지원합니다.
- 상태 점검은 `serve get`이 가장 자세합니다.
- 추론 endpoint의 모델 이름은 플러그인 이름이 아니라 serving spec의 `metadata.name`입니다.
- 루트 통합 예제는 [docker/Dockerfile](/home/scouter/proj_2026_1_etri/test/docker/Dockerfile) 기반 이미지와 `kfp-workflow:latest`를 기본값으로 사용하지만, predictor 이미지는 스펙으로 바꿀 수 있습니다.
