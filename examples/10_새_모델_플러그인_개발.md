# 10. 새 모델 플러그인 개발

## 개요

kfp-workflow의 플러그인 시스템은 새로운 모델을 파이프라인에 통합하는 표준화된 방법을 제공합니다. `ModelPlugin` 추상 기반 클래스(ABC)를 구현하기만 하면, 기존의 파이프라인 컴파일러, CLI, 서빙 인프라를 그대로 활용할 수 있습니다.

이 문서는 플러그인 아키텍처의 설계 원칙부터 구현, 등록, 설정, 테스트, 서빙, Docker 패키징, HPO 지원까지 전체 과정을 다룹니다. 기존 `mambasl-cmapss` 플러그인의 실제 코드를 참조 구현으로 사용합니다.

## 아키텍처 원칙

### 의존성 방향

```
kfp_workflow (프레임워크) ─── 오케스트레이션, 인터페이스 정의
  │
  ├── plugins/base.py      → ModelPlugin ABC, Result 데이터클래스 정의
  ├── components/*.py       → KFP 컴포넌트가 플러그인을 호출
  ├── pipeline/compiler.py  → 파이프라인 DAG 조립 및 KFP 컴파일
  ├── serving/predictor.py  → KServe 예측기가 플러그인 serving hooks 호출
  └── tune/engine.py        → HPO 엔진이 plugin.hpo_objective() 호출

플러그인 어댑터 (kfp_workflow/plugins/mambasl_cmapss.py)
  │ import
  ▼
모델 패키지 (models/mambasl-new/) ─── 순수 ML 로직
```

**핵심 규칙:**
- **프레임워크**는 인터페이스를 정의하고 오케스트레이션을 소유합니다
- **플러그인 어댑터**는 프레임워크 인터페이스를 구현하고, 모델 패키지의 ML 함수를 호출합니다
- **모델 패키지**는 프레임워크를 임포트하지 않습니다 — 순수한 ML 코드입니다
- 플러그인 어댑터 파일은 `src/kfp_workflow/plugins/` 안에 위치하지만, 무거운 임포트는 메서드 내부에서 지연 로딩합니다

### 관심사 분리

| 계층 | 책임 | 예시 |
|------|------|------|
| **프레임워크** | 파이프라인 DAG, KFP 컴파일, CLI, 서빙 CRUD, HPO 엔진 | `compiler.py`, `engine.py` |
| **플러그인 어댑터** | 프레임워크 ↔ 모델 패키지 간 변환, 설정 매핑 | `mambasl_cmapss.py` |
| **모델 패키지** | 데이터 로딩, 전처리, 모델 아키텍처, 학습 루프, 평가 | `mambasl_new/cmapss/` |

### 데이터 흐름

```
KFP 파이프라인 컴포넌트 (Kubernetes Pod)
  │
  │  1. spec_json을 JSON으로 역직렬화
  │  2. get_plugin(spec["model"]["name"]) 으로 플러그인 인스턴스 생성
  │  3. 플러그인 메서드 호출
  │  4. Result 데이터클래스를 JSON으로 직렬화하여 다음 컴포넌트로 전달
  │
  ▼
  무거운 데이터 (numpy 배열):
    → PVC의 .npy 파일로 저장
    → 경로만 JSON으로 전달
```

## ModelPlugin ABC

모든 플러그인은 `ModelPlugin` ABC를 구현해야 합니다. 파이프라인의 각 스테이지에 대응하는 메서드와 추론 메서드로 구성됩니다.

### 파이프라인 스테이지

```
┌──────────┐    ┌────────────┐    ┌─────────┐    ┌──────────┐    ┌────────────┐
│ load_data│───>│ preprocess │───>│  train  │───>│ evaluate │───>│ save_model │
└──────────┘    └────────────┘    └─────────┘    └──────────┘    └────────────┘
LoadDataResult  PreprocessResult  TrainResult    EvalResult      SaveResult
```

각 스테이지는 독립적인 Kubernetes Pod에서 실행됩니다. 스테이지 간에는 JSON 문자열만 전달되므로, 무거운 데이터는 반드시 PVC에 파일로 저장하고 경로만 Result에 포함해야 합니다.

### 메서드 시그니처 (필수)

```python
from kfp_workflow.plugins.base import (
    ModelPlugin, LoadDataResult, PreprocessResult,
    TrainResult, EvalResult, SaveResult,
)

class MyPlugin(ModelPlugin):

    @staticmethod
    def name() -> str:
        """플러그인 고유 식별자 (spec의 model.name과 일치)."""
        ...

    def load_data(
        self,
        spec: Dict[str, Any],       # 전체 PipelineSpec (JSON→dict)
        data_mount_path: str,        # PVC 데이터 마운트 경로
    ) -> LoadDataResult:
        """원시 데이터를 로드하거나 다운로드합니다."""
        ...

    def preprocess(
        self,
        spec: Dict[str, Any],
        load_result: LoadDataResult,  # 이전 스테이지 출력
        artifacts_dir: str,           # .npy 파일 저장 디렉토리 (PVC 경로)
    ) -> PreprocessResult:
        """원시 데이터를 학습용 numpy 배열로 변환합니다."""
        ...

    def train(
        self,
        spec: Dict[str, Any],
        preprocess_result: PreprocessResult,
        model_dir: str,               # 모델 가중치 저장 디렉토리 (PVC 경로)
    ) -> TrainResult:
        """모델을 학습하고 최적 가중치를 저장합니다."""
        ...

    def evaluate(
        self,
        spec: Dict[str, Any],
        train_result: TrainResult,
        preprocess_result: PreprocessResult,
    ) -> EvalResult:
        """테스트셋에서 모델을 평가합니다."""
        ...

    def save_model(
        self,
        spec: Dict[str, Any],
        train_result: TrainResult,
        eval_result: EvalResult,
        final_model_dir: str,          # 최종 모델 저장 경로 (PVC)
    ) -> SaveResult:
        """모델을 최종 경로에 복사하고 레지스트리에 등록합니다."""
        ...

    def predict(
        self,
        model_path: str,              # 저장된 모델 아티팩트 경로
        input_data: Any,              # numpy 배열 [batch, seq_len, features]
        model_config: Dict[str, Any], # model_config.json 내용
    ) -> Any:
        """KServe 서빙용 추론을 수행합니다."""
        ...
```

## `spec` 딕셔너리 구조

모든 플러그인 메서드의 첫 번째 인자 `spec`은 전체 `PipelineSpec`(또는 `TuneSpec`)을 JSON으로 직렬화한 후 `json.loads()`로 역직렬화한 Python dict입니다. 프레임워크가 컴포넌트에 전달하는 과정:

```python
# pipeline/compiler.py — 컴파일 시
spec_json = spec.model_dump_json()

# components/load_data.py — 실행 시
spec = json.loads(spec_json)
plugin = get_plugin(spec["model"]["name"])
result = plugin.load_data(spec=spec, data_mount_path=spec["storage"]["data_mount_path"])
```

따라서 플러그인은 Pydantic 모델이 아닌 일반 Python dict를 받습니다. 자주 접근하는 경로:

```python
# 모델 아키텍처 설정
spec["model"]["name"]           # "mambasl-cmapss"
spec["model"]["config"]         # {"d_model": 32, "d_state": 8, ...}
spec["model"]["version"]        # "v1"

# 데이터셋 설정
spec["dataset"]["name"]         # "cmapss"
spec["dataset"]["config"]       # {"fd_name": "FD001", ...}

# 학습 하이퍼파라미터
spec["train"]["batch_size"]     # 64
spec["train"]["learning_rate"]  # 0.001
spec["train"]["max_epochs"]     # 50
spec["train"]["patience"]       # 8
spec["train"]["seed"]           # 42
spec["train"]["val_split"]      # 0.2
spec["train"]["selection_metric"]  # "rmse"

# 런타임 설정
spec["runtime"]["use_gpu"]      # false
spec["runtime"]["torch_num_threads"]  # 4

# 스토리지 경로
spec["storage"]["data_mount_path"]   # "/mnt/data"
spec["storage"]["model_mount_path"]  # "/mnt/models"
```

## Result 데이터클래스

각 스테이지는 프레임워크가 정의한 데이터클래스를 반환해야 합니다. 이 데이터클래스들은 `result_to_dict()`로 직렬화되어 다음 스테이지에 JSON 문자열로 전달됩니다.

### LoadDataResult

```python
@dataclass
class LoadDataResult:
    data_dir: str            # 원시 데이터 디렉토리 경로 (PVC 또는 로컬)
    dataset_name: str        # 데이터셋 식별자 (예: "FD001")
    num_train_samples: int   # 학습 샘플 수
    num_test_samples: int    # 테스트 샘플 수
    metadata: Dict[str, Any] = field(default_factory=dict)
```

**metadata 활용 예시** (mambasl-cmapss):
```python
metadata={
    "fd_config": {"n_conditions": 6, "n_units_train": 100},
    "rul_test_count": 100,
}
```

### PreprocessResult

```python
@dataclass
class PreprocessResult:
    artifacts_dir: str       # .npy 파일이 저장된 디렉토리
    x_train_path: str        # 학습 입력 경로 (.npy)
    y_train_path: str        # 학습 레이블 경로 (.npy)
    x_val_path: str          # 검증 입력 경로 (.npy)
    y_val_path: str          # 검증 레이블 경로 (.npy)
    x_test_path: str         # 테스트 입력 경로 (.npy)
    y_test_path: str         # 테스트 레이블 경로 (.npy)
    feature_dim: int         # 입력 특성 차원 (c_in)
    seq_len: int             # 시퀀스 / 윈도우 길이
    num_train: int           # 학습 윈도우 수
    num_val: int             # 검증 윈도우 수
    num_test: int            # 테스트 윈도우 수
    metadata: Dict[str, Any] = field(default_factory=dict)
```

**핵심 계약**: preprocess는 6개의 `.npy` 파일을 `artifacts_dir`에 저장하고, 그 경로를 Result에 포함해야 합니다. 이후 스테이지에서 `np.load()`로 직접 로드합니다.

## 단계별 구현 가이드

### 1단계: 모델 패키지 생성

새 모델의 ML 로직을 `models/` 디렉토리에 독립 Python 패키지로 생성합니다. 이 패키지는 kfp-workflow를 임포트하지 않는 순수 ML 코드입니다.

```
models/
└── my-new-model/
    ├── pyproject.toml
    ├── README.md
    └── src/
        └── my_new_model/
            ├── __init__.py
            ├── model.py         # nn.Module 정의
            ├── data.py          # 데이터 다운로드/로딩
            ├── preprocess.py    # 정규화, 윈도잉, 분할
            └── train.py         # 학습 루프, 평가 함수
```

`pyproject.toml`:

```toml
[build-system]
requires = ["setuptools>=68.0", "wheel"]
build-backend = "setuptools.build_meta"

[project]
name = "my-new-model"
version = "0.1.0"
requires-python = ">=3.10"
dependencies = [
    "torch>=2.4",
    "numpy>=1.24",
]

[tool.setuptools]
package-dir = {"" = "src"}

[tool.setuptools.packages.find]
where = ["src"]
```

### 2단계: 플러그인 어댑터 작성

`src/kfp_workflow/plugins/` 디렉토리에 어댑터 파일을 생성합니다. 어댑터는 프레임워크의 인터페이스를 구현하고, 모델 패키지의 함수를 지연 임포트로 호출합니다.

#### 지연 임포트 패턴

무거운 의존성 (torch, numpy, 모델 패키지)은 **메서드 내부에서 임포트**합니다. 이렇게 하면 CLI가 시작될 때 모든 모델의 무거운 라이브러리를 로드하지 않습니다:

```python
# ✓ 올바른 패턴 — 지연 임포트
def load_data(self, spec, data_mount_path):
    from my_new_model.data import download_and_load  # 호출 시에만 로드
    ...

# ✗ 잘못된 패턴 — 모듈 레벨 임포트
import torch                                          # CLI 시작 시 항상 로드
from my_new_model.data import download_and_load
```

#### 설정 병합 헬퍼 (`_build_cfg` 패턴)

mambasl-cmapss 플러그인은 `spec["model"]["config"]`과 `spec["train"]`의 값을 모델 패키지가 기대하는 단일 flat dict로 병합하는 `_build_cfg()` 헬퍼를 사용합니다:

```python
def _build_cfg(spec: Dict[str, Any]) -> Dict[str, Any]:
    """model.config + train 파라미터를 모델 패키지가 기대하는 flat dict로 병합."""
    cfg: Dict[str, Any] = dict(spec["model"].get("config", {}))
    train = spec.get("train", {})

    # spec 키 이름 → 모델 패키지 키 이름 매핑
    cfg.setdefault("batch_size", train.get("batch_size", 64))
    cfg.setdefault("lr", train.get("learning_rate", 1e-3))

    # 모든 필수 키에 기본값 보장
    cfg.setdefault("hidden_dim", 128)
    cfg.setdefault("num_layers", 4)
    return cfg
```

이 패턴은 프레임워크의 spec 구조와 모델 패키지의 설정 형식 사이의 변환 계층으로 기능합니다.

#### 전체 어댑터 예시

```python
# src/kfp_workflow/plugins/my_new_model.py
"""My New Model plugin adapter."""

from __future__ import annotations

import json
import shutil
from pathlib import Path
from typing import Any, Dict, Optional, Type

from pydantic import BaseModel, ConfigDict

from kfp_workflow.plugins.base import (
    EvalResult,
    LoadDataResult,
    ModelPlugin,
    PreprocessResult,
    SaveResult,
    TrainResult,
)


# ---------------------------------------------------------------------------
# Config 스키마 (선택이지만 --set 검증에 권장)
# ---------------------------------------------------------------------------

class MyModelConfig(BaseModel):
    """model.config 검증 스키마."""
    hidden_dim: int = 128
    num_layers: int = 4
    dropout: float = 0.1

    model_config = ConfigDict(extra="allow")


class MyDatasetConfig(BaseModel):
    """dataset.config 검증 스키마."""
    dataset_name: str = "default"
    split_ratio: float = 0.8

    model_config = ConfigDict(extra="allow")


# ---------------------------------------------------------------------------
# 설정 병합 헬퍼
# ---------------------------------------------------------------------------

def _build_cfg(spec: Dict[str, Any]) -> Dict[str, Any]:
    """spec을 모델 패키지가 기대하는 flat dict로 변환."""
    cfg: Dict[str, Any] = dict(spec["model"].get("config", {}))
    train = spec.get("train", {})
    cfg.setdefault("batch_size", train.get("batch_size", 64))
    cfg.setdefault("lr", train.get("learning_rate", 1e-3))
    cfg.setdefault("weight_decay", train.get("weight_decay", 1e-4))
    cfg.setdefault("hidden_dim", 128)
    cfg.setdefault("num_layers", 4)
    cfg.setdefault("dropout", 0.1)
    return cfg


# ---------------------------------------------------------------------------
# 플러그인 구현
# ---------------------------------------------------------------------------

class MyNewModelPlugin(ModelPlugin):
    """My New Model 플러그인."""

    @staticmethod
    def name() -> str:
        return "my-new-model"

    @classmethod
    def model_config_schema(cls) -> Optional[Type]:
        return MyModelConfig

    @classmethod
    def dataset_config_schema(cls) -> Optional[Type]:
        return MyDatasetConfig

    # -- Stage 1: load_data ------------------------------------------------

    def load_data(
        self,
        spec: Dict[str, Any],
        data_mount_path: str,
    ) -> LoadDataResult:
        from my_new_model.data import download_and_load

        ds_cfg = spec["dataset"].get("config", {})
        data_dir = str(
            Path(data_mount_path)
            / ds_cfg.get("dataset_name", "default")
        )

        train_count, test_count = download_and_load(data_dir)

        return LoadDataResult(
            data_dir=data_dir,
            dataset_name=ds_cfg.get("dataset_name", "default"),
            num_train_samples=train_count,
            num_test_samples=test_count,
        )

    # -- Stage 2: preprocess -----------------------------------------------

    def preprocess(
        self,
        spec: Dict[str, Any],
        load_result: LoadDataResult,
        artifacts_dir: str,
    ) -> PreprocessResult:
        import numpy as np
        from my_new_model.preprocess import prepare_arrays

        out = Path(artifacts_dir)
        out.mkdir(parents=True, exist_ok=True)

        arrays = prepare_arrays(
            data_dir=load_result.data_dir,
            config=spec.get("dataset", {}).get("config", {}),
        )

        # .npy 파일로 저장 — 무거운 데이터는 PVC에 저장하고 경로만 전달
        paths = {}
        for arr_name, arr in arrays.items():
            p = out / f"{arr_name}.npy"
            np.save(str(p), arr)
            paths[arr_name] = str(p)

        return PreprocessResult(
            artifacts_dir=str(out),
            x_train_path=paths["x_train"],
            y_train_path=paths["y_train"],
            x_val_path=paths["x_val"],
            y_val_path=paths["y_val"],
            x_test_path=paths["x_test"],
            y_test_path=paths["y_test"],
            feature_dim=int(arrays["x_train"].shape[-1]),
            seq_len=int(arrays["x_train"].shape[1]),
            num_train=int(arrays["x_train"].shape[0]),
            num_val=int(arrays["x_val"].shape[0]),
            num_test=int(arrays["x_test"].shape[0]),
        )

    # -- Stage 3: train ----------------------------------------------------

    def train(
        self,
        spec: Dict[str, Any],
        preprocess_result: PreprocessResult,
        model_dir: str,
    ) -> TrainResult:
        import numpy as np
        import torch
        from my_new_model.train import train_loop

        cfg = _build_cfg(spec)
        train_cfg = spec.get("train", {})

        # .npy에서 배열 로드
        x_train = np.load(preprocess_result.x_train_path)
        y_train = np.load(preprocess_result.y_train_path)
        x_val = np.load(preprocess_result.x_val_path)
        y_val = np.load(preprocess_result.y_val_path)

        # 시드 고정
        np.random.seed(train_cfg.get("seed", 42))
        torch.manual_seed(train_cfg.get("seed", 42))

        model, best_epoch, train_loss, val_loss = train_loop(
            x_train, y_train, x_val, y_val,
            model_cfg=cfg,
            max_epochs=train_cfg.get("max_epochs", 50),
            patience=train_cfg.get("patience", 8),
            lr=cfg["lr"],
        )

        # 모델 가중치 저장
        out = Path(model_dir)
        out.mkdir(parents=True, exist_ok=True)
        model_path = str(out / "model.pt")
        torch.save(
            {k: v.detach().cpu() for k, v in model.state_dict().items()},
            model_path,
        )

        # model_config.json — predict()에서 모델을 재구성할 때 필요
        config_path = out / "model_config.json"
        config_path.write_text(json.dumps({
            "cfg": cfg,
            "feature_dim": preprocess_result.feature_dim,
            "seq_len": preprocess_result.seq_len,
        }, indent=2))

        return TrainResult(
            model_path=model_path,
            best_epoch=best_epoch,
            train_loss=train_loss,
            val_loss=val_loss,
        )

    # -- Stage 4: evaluate -------------------------------------------------

    def evaluate(
        self,
        spec: Dict[str, Any],
        train_result: TrainResult,
        preprocess_result: PreprocessResult,
    ) -> EvalResult:
        import numpy as np
        import torch
        from my_new_model.train import evaluate_model

        cfg = _build_cfg(spec)
        x_test = np.load(preprocess_result.x_test_path)
        y_test = np.load(preprocess_result.y_test_path)

        metrics = evaluate_model(
            model_path=train_result.model_path,
            x_test=x_test,
            y_test=y_test,
            model_cfg=cfg,
        )

        return EvalResult(
            metrics=metrics,
            model_path=train_result.model_path,
        )

    # -- Stage 5: save_model -----------------------------------------------

    def save_model(
        self,
        spec: Dict[str, Any],
        train_result: TrainResult,
        eval_result: EvalResult,
        final_model_dir: str,
    ) -> SaveResult:
        from kfp_workflow.registry.model_registry import FileModelRegistry

        model_name = spec["model"]["name"]
        model_version = spec["model"].get("version", "v1")

        final = Path(final_model_dir)
        final.mkdir(parents=True, exist_ok=True)

        # model.pt 복사
        src_pt = Path(train_result.model_path)
        dst_pt = final / "model.pt"
        if src_pt.resolve() != dst_pt.resolve():
            shutil.copy2(str(src_pt), str(dst_pt))

        # model_config.json 복사 (서빙에 필요)
        src_cfg = src_pt.parent / "model_config.json"
        dst_cfg = final / "model_config.json"
        if src_cfg.exists() and src_cfg.resolve() != dst_cfg.resolve():
            shutil.copy2(str(src_cfg), str(dst_cfg))

        # 모델 레지스트리에 등록
        registry_path = str(
            Path(spec["storage"]["model_mount_path"]) / ".model_registry.json"
        )
        try:
            registry = FileModelRegistry(registry_path=registry_path)
            registry.register_model(
                name=model_name,
                version=model_version,
                uri=str(dst_pt),
                framework="pytorch",
                description=f"My New Model {spec['dataset']['name']}",
                parameters={
                    "metrics": eval_result.metrics,
                    "config": _build_cfg(spec),
                },
            )
        except Exception:
            pass  # 클러스터 외부에서는 레지스트리 쓰기 실패 가능 — 치명적이지 않음

        return SaveResult(
            saved_path=str(dst_pt),
            model_name=model_name,
            model_version=model_version,
        )

    # -- Inference ----------------------------------------------------------

    def predict(
        self,
        model_path: str,
        input_data: Any,
        model_config: Dict[str, Any],
    ) -> Any:
        import torch
        from my_new_model.model import MyModel

        # model_config.json에서 로드된 설정으로 모델 재구성
        cfg = model_config.get("cfg", model_config)
        model = MyModel(**cfg)
        state = torch.load(model_path, map_location="cpu", weights_only=True)
        model.load_state_dict(state)
        model.eval()

        with torch.no_grad():
            tensor = torch.tensor(input_data, dtype=torch.float32)
            return model(tensor).numpy()
```

### 4단계: Config 스키마 정의

Config 스키마는 `--set` 오버라이드의 타입 검증에 사용됩니다. `spec validate` 명령에서 플러그인의 스키마가 있으면 `model.config`와 `dataset.config`에 대한 추가 검증을 수행합니다.

#### 설계 지침

**`extra="allow"` 사용:**

```python
class MyModelConfig(BaseModel):
    hidden_dim: int = 128
    num_layers: int = 4
    model_config = ConfigDict(extra="allow")  # 알 수 없는 필드 허용
```

이유:
- 알려진 필드는 타입 검증 (예: `hidden_dim`에 문자열 전달 시 경고)
- 알 수 없는 필드는 경고만 표시 (오류 아님)
- 실험적 파라미터를 스키마 변경 없이 전달 가능

**모든 필드에 기본값 제공:**

스키마의 모든 필드에 기본값을 제공하면, 최소한의 스펙으로도 동작합니다. `model.config`를 아예 생략해도 기본값으로 동작:

```yaml
# 이것만으로도 유효
model:
  name: my-new-model
```

## KServe 서빙 통합

KServe 커스텀 예측기는 플러그인 serving hooks를 사용합니다. 서빙 시 동작 흐름:

```
클라이언트 (HTTP POST)
  │  {"instances": [[[0.1, 0.2, ...], ...]]}
  │
  ▼
PluginPredictor (kfp_workflow.serving.predictor)
  │
  │  1. model_dir에서 model_config.json 로드
  │  2. get_plugin(MODEL_PLUGIN_NAME)으로 플러그인 인스턴스화
  │  3. plugin.serving_model_filenames()로 아티팩트 파일명 결정
  │  4. plugin.load_serving_artifact(...) / warmup_serving_artifact(...) 수행
  │  5. 요청마다 plugin.predict_loaded(...) 호출
  │  6. numpy 결과를 JSON으로 변환하여 반환
  │
  ▼
클라이언트 ← {"predictions": [24.5, 31.2, ...]}
```

환경 변수 (InferenceService 매니페스트에 자동 설정):
- `MODEL_PLUGIN_NAME`: 플러그인 이름 (예: `my-new-model`)
- `MODEL_DIR`: 모델 디렉토리 경로 (PVC 마운트 경로)
- `MODEL_NAME`: KServe 모델 이름

`model_config.json`은 서빙 시 모델 재구성/아티팩트 로드에 사용됩니다. 따라서 `train()` 또는 `save_model()` 단계에서 서빙에 필요한 모든 정보를 함께 저장해야 합니다:

```python
# train()에서 model_config.json 저장
config_path.write_text(json.dumps({
    "cfg": cfg,              # 모델 아키텍처 설정
    "feature_dim": 14,       # 입력 차원 (모델 재구성에 필요)
    "seq_len": 30,           # 시퀀스 길이 (모델 재구성에 필요)
}, indent=2))

# predict_loaded()에서 model_config.json 사용
def predict_loaded(self, artifact, input_data, model_config):
    cfg = model_config.get("cfg", model_config)
    c_in = model_config.get("feature_dim", input_data.shape[2])
    # ... 모델 재구성 ...
```

기본 아티팩트명이 `model.pt`가 아닌 플러그인은 `serving_model_filenames()`를 구현해 후보 파일명을 반환해야 합니다. 예를 들어 joblib 기반 모델이라면 `["model.joblib"]`처럼 명시합니다.

## 프레임워크 호출 흐름 상세

플러그인 개발 시 프레임워크가 각 메서드를 **어떻게** 호출하는지 이해하면, 인터페이스 계약을 정확히 지킬 수 있습니다.

### 파이프라인 컴포넌트 디스패치

각 KFP 컴포넌트는 동일한 패턴으로 플러그인을 호출합니다:

```python
# components/load_data.py (실제 코드)
@dsl.component(base_image=BASE_IMAGE)
def load_data_component(spec_json: str) -> str:
    import json
    from kfp_workflow.plugins import get_plugin
    from kfp_workflow.plugins.base import result_to_dict

    spec = json.loads(spec_json)
    plugin = get_plugin(spec["model"]["name"])    # ← 플러그인 인스턴스화
    result = plugin.load_data(                     # ← 메서드 호출
        spec=spec,
        data_mount_path=spec["storage"]["data_mount_path"],
    )
    return json.dumps(result_to_dict(result))      # ← Result → JSON
```

**이전 스테이지 출력을 재구성하여 전달:**

```python
# components/train.py (실제 코드)
@dsl.component(base_image=BASE_IMAGE)
def train_component(spec_json: str, preprocess_result_json: str) -> str:
    import json
    from kfp_workflow.plugins import get_plugin
    from kfp_workflow.plugins.base import PreprocessResult, result_to_dict

    spec = json.loads(spec_json)
    pp_raw = json.loads(preprocess_result_json)
    preprocess_result = PreprocessResult(**pp_raw)  # ← JSON에서 데이터클래스 복원

    plugin = get_plugin(spec["model"]["name"])
    model_dir = (
        f"{spec['storage']['model_mount_path']}"
        f"/{spec['model']['name']}/{spec['model'].get('version', 'v1')}"
    )
    result = plugin.train(
        spec=spec,
        preprocess_result=preprocess_result,
        model_dir=model_dir,
    )
    return json.dumps(result_to_dict(result))
```

### PVC 경로 규칙

프레임워크가 각 스테이지에 전달하는 경로:

| 경로 | 결정 주체 | 계산식 |
|------|-----------|--------|
| `data_mount_path` | spec | `spec.storage.data_mount_path` (예: `/mnt/data`) |
| `artifacts_dir` | 컴포넌트 | `{model_mount_path}/artifacts/{model_name}/{model_version}` |
| `model_dir` | 컴포넌트 | `{model_mount_path}/{model_name}/{model_version}` |
| `final_model_dir` | 컴포넌트 | `{model_mount_path}/{model_name}/{model_version}` |

플러그인은 이 경로를 그대로 사용하면 됩니다. 디렉토리 생성은 플러그인 책임:

```python
out = Path(artifacts_dir)
out.mkdir(parents=True, exist_ok=True)
```

## HPO (하이퍼파라미터 최적화) 지원

플러그인은 선택적으로 HPO를 지원할 수 있습니다. 프로젝트가 HPO 오케스트레이션을 소유하고, 플러그인은 프로젝트가 정의한 타입과 인터페이스를 따릅니다.

### 의존성 원칙

```
kfp_workflow (프로젝트) — HPO 오케스트레이션 소유
  │
  ├── specs.py          → SearchParamSpec (프로젝트 타입)
  ├── tune/engine.py    → Optuna 스터디, 시행 루프, 파라미터 제안
  ├── tune/katib.py     → Katib CRD 매니페스트 생성
  ├── tune/exceptions.py → TrialPruned 예외
  └── plugins/base.py   → HPO 계약 (3개 선택적 메서드)

플러그인 — 프로젝트 타입만 사용 (Optuna 직접 임포트 금지)
  │
  ├── hpo_search_space()  → List[SearchParamSpec] 반환
  ├── hpo_base_config()   → Dict 반환
  └── hpo_objective()     → 단일 시행 실행, float 반환
                             프루닝 시 TrialPruned 발생
```

**핵심 규칙**: 플러그인은 `optuna`를 직접 임포트하지 않습니다. `SearchParamSpec`과 `TrialPruned`만 프로젝트에서 임포트합니다.

### HPO 메서드

| 메서드 | 호출 주체 | 반환 | 역할 |
|--------|-----------|------|------|
| `hpo_search_space(spec, profile)` | `tune/engine.py` | `List[SearchParamSpec]` | 내장 탐색 공간 정의 |
| `hpo_base_config(spec)` | `tune/engine.py` | `Dict[str, Any]` | 검색 파라미터 외의 고정 설정 |
| `hpo_objective(spec, params, data_mount_path)` | `tune/engine.py` | `float` | 한 번의 학습+평가, 목적 메트릭 반환 |

기본 구현은 `NotImplementedError`를 발생시킵니다. HPO를 지원하지 않는 플러그인은 이 메서드들을 구현하지 않아도 됩니다.

### 구현 예시

```python
from kfp_workflow.specs import SearchParamSpec
from kfp_workflow.tune.exceptions import TrialPruned


class MyNewModelPlugin(ModelPlugin):
    # ... 기존 메서드 생략 ...

    def hpo_search_space(
        self, spec: Dict[str, Any], profile: str,
    ) -> list:
        from kfp_workflow.specs import SearchParamSpec

        if profile == "default":
            return [
                SearchParamSpec(
                    name="train.learning_rate",
                    type="log_float",
                    low=1e-4, high=1e-2,
                ),
                SearchParamSpec(
                    name="model.config.hidden_dim",
                    type="categorical",
                    values=[64, 128, 256],
                ),
                SearchParamSpec(
                    name="train.batch_size",
                    type="int",
                    low=16, high=128, step=16,
                ),
            ]
        elif profile == "aggressive":
            return [
                SearchParamSpec(
                    name="train.learning_rate",
                    type="log_float",
                    low=1e-5, high=1e-1,
                ),
                SearchParamSpec(
                    name="model.config.hidden_dim",
                    type="categorical",
                    values=[32, 64, 128, 256, 512],
                ),
                SearchParamSpec(
                    name="model.config.num_layers",
                    type="int",
                    low=2, high=8,
                ),
                SearchParamSpec(
                    name="model.config.dropout",
                    type="float",
                    low=0.0, high=0.5, step=0.1,
                ),
                SearchParamSpec(
                    name="train.batch_size",
                    type="categorical",
                    values=[16, 32, 64, 128, 256],
                ),
                SearchParamSpec(
                    name="train.weight_decay",
                    type="log_float",
                    low=1e-6, high=1e-2,
                ),
            ]
        return []

    def hpo_base_config(self, spec: Dict[str, Any]) -> Dict[str, Any]:
        return _build_cfg(spec)

    def hpo_objective(
        self,
        spec: Dict[str, Any],
        params: Dict[str, Any],
        data_mount_path: str,
    ) -> float:
        from my_new_model.train import train_and_evaluate
        from kfp_workflow.tune.exceptions import TrialPruned

        merged = {**self.hpo_base_config(spec), **params}

        try:
            metric = train_and_evaluate(
                data_path=data_mount_path,
                config=merged,
                max_epochs=spec.get("train", {}).get("max_epochs", 50),
            )
        except ValueError as e:
            raise TrialPruned(str(e))

        return metric  # 낮을수록 좋음 (예: RMSE)
```

### SearchParamSpec 타입

| 타입 | 설명 | 필수 필드 | 예시 |
|------|------|----------|------|
| `categorical` | 후보 값 목록에서 선택 | `values` | `values=[32, 64, 128]` |
| `int` | 정수 범위 | `low`, `high`, (`step`) | `low=16, high=128, step=16` |
| `float` | 실수 범위 | `low`, `high`, (`step`) | `low=0.0, high=0.5, step=0.1` |
| `log_float` | 로그 스케일 실수 | `low`, `high` | `low=1e-5, high=1e-1` |

### 데이터 캐싱 최적화

HPO에서는 동일한 데이터를 여러 시행에 걸쳐 반복 로드합니다. 플러그인 인스턴스에 캐시를 두면 성능이 크게 향상됩니다.

mambasl-cmapss 플러그인의 실제 패턴:

```python
class MyNewModelPlugin(ModelPlugin):

    def hpo_objective(self, spec, params, data_mount_path):
        # 캐시 키 — 데이터 변형에 영향을 주는 파라미터로 구성
        cache_key = (
            spec["dataset"]["config"].get("dataset_name"),
            int(params.get("window_size", 50)),
        )

        if not hasattr(self, "_hpo_cache") or self._hpo_cache_key != cache_key:
            # 데이터 로드 & 전처리 (첫 시행 또는 캐시 키 변경 시)
            self._hpo_cache = self._load_and_preprocess(spec, params, data_mount_path)
            self._hpo_cache_key = cache_key

        # 캐시된 데이터로 학습
        return self._train_single_trial(self._hpo_cache, params, spec)
```

**캐시 키 설계 시 주의**: 데이터 전처리 결과에 영향을 주는 파라미터(예: `window_size`, `max_rul`)를 캐시 키에 포함해야 합니다. 검색 공간에 이런 파라미터가 포함되면 해당 값이 바뀔 때 자동으로 캐시가 무효화됩니다.

## 참조 구현: mambasl-cmapss 분석

기존 `mambasl-cmapss` 플러그인(`src/kfp_workflow/plugins/mambasl_cmapss.py`)의 주요 설계 결정을 분석합니다.

### 설정 병합: `_build_cfg()`

```python
def _build_cfg(spec: Dict[str, Any]) -> Dict[str, Any]:
    cfg: Dict[str, Any] = dict(spec["model"].get("config", {}))
    train = spec.get("train", {})
    # 키 이름 변환: learning_rate → lr
    cfg.setdefault("batch_size", train.get("batch_size", 64))
    cfg.setdefault("lr", train.get("learning_rate", 1e-3))
    cfg.setdefault("weight_decay", train.get("weight_decay", 1e-4))
    # 17개의 기본값 보장
    cfg.setdefault("d_model", 64)
    cfg.setdefault("d_state", 16)
    # ...
    return cfg
```

이 함수가 하는 일:
1. `model.config`의 모든 값을 flat dict에 복사
2. `train` 섹션의 하이퍼파라미터를 모델 패키지가 기대하는 키 이름으로 매핑 (`learning_rate` → `lr`)
3. 누락된 모든 키에 기본값을 `setdefault`로 설정

`_build_cfg`는 `train()`, `evaluate()`, `hpo_base_config()` 모두에서 사용됩니다.

cache_key = (
